### 1.3 文献核心发现

**Narrative Theory-Driven LLM Methods（综述论文）**：
- 提出了反映叙事学区分的叙事任务分类体系
- 强调LLM使NLP流水线与抽象叙事概念之间易于连接
- 挑战：缺乏统一的"叙事质量"评估基准

**NarrativeLoom**：
- 基于Campbell的盲变异与选择性保留理论的多人物系统
- AI人物生成多样化选项（盲变异）；用户选择并优化（选择性保留）
- 结果：故事在所有Torrance测试维度上得分更高（流畅性、灵活性、原创性、精致性）

**ConStory-Bench**：
- 定义了5大类错误和19个子类型用于一致性bug检测
- 发现错误最常见于事实和时间维度
- 错误倾向于出现在叙事的中段

## 2. 对比分析

### 2.1 架构对比

| 维度 | 现有研究 | 我们的方案 |
|------|----------|------------|
| **目标领域** | 通用故事生成 | **网文（商业文学）** |
| **架构设计** | 单层或双层 | **四层瀑布模型** |
| **协作模式** | 多Agent生成选项 | **专家会议+主编决策** |
| **理论基础** | 通用叙事学 | **叙事学+网文商业理论（双库映射）** |
| **工程化程度** | 较弱 | **软件工程思维（分层解耦、RAG、向量数据库）** |

### 2.2 与NarrativeLoom的详细对比

| 方面 | NarrativeLoom | 我们的L2架构层 |
|------|---------------|----------------|
| **理论依据** | Campbell的BVSR理论（盲变异→选择性保留） | 专家会议（架构师+编辑+人物设计师） |
| **Agent角色** | 通用AI人物 | **专业化的专家，职责明确** |
| | | - 剧情架构师：序列/功能拆解 |
| | | - 网络编辑：市场卖点、节奏、爽点 |
| | | - 人物设计师：行动元分配、OOC预防 |
| **用户角色** | 创意总监（选择/优化） | 主编（最终决策、逻辑审计） |
| **迭代方式** | 单轮选择 | **多轮专家讨论（最多3轮迭代）** |
| **领域知识** | 通用叙事 | **网文专属（爽点公式、黄金三章、毒点规避）** |

### 2.3 与StorySage的详细对比

| 方面 | StorySage | 我们的方案 |
|------|-----------|------------|
| **任务类型** | 自传写作 | **小说/网文创作** |
| **Agent组成** | 采访者、记录员、规划师、撰写者、协调者 | **L1-L4分层Agent** |
| **记忆机制** | 基于会话的记忆收集 | **共享设定数据库+状态机** |
| **输出产物** | 单一自传文档 | **结构化流水线：愿景→大纲→章纲→正文** |

### 2.4 我们的独特贡献

1. **领域专业化**：首个专门针对中文网文创作的系统，融入商业网文理论（爽点、节奏、黄金三章）

2. **四层架构模型**：
   ```
   L1 种子层     → 故事愿景文档
   L2 架构层     → 精修故事大纲（专家会议）
   L3 叙事层     → 章纲/细纲（效果→技术映射）
   L4 渲染层     → 最终正文
   ```

3. **双数据库映射机制**：
   - 网文理论库（效果学）："要达到什么效果"
   - 叙事学理论库（结构学）："技术上如何实现"
   - 映射关系库：将抽象效果翻译为具体叙事技术

4. **专家会议协议**：
   - 三位专业专家，各自拥有专属RAG数据库
   - 三种驱动模式：人物驱动/剧情驱动/市场驱动
   - 用户担任主编，拥有最终决策权

5. **多维度切片的RAG方案**：
   - 剧情维度：按序列/事件切分
   - 人物维度：按人物出场切分
   - 情绪维度：按情绪阶段切分
   - 功能维度：按叙事功能切分

## 3. arXiv发表可行性分析

### 3.1 适合的分类

arXiv适合发表此类研究。推荐分类：

| 主分类 | 次分类 | 可选分类 |
|--------|--------|----------|
| `cs.CL`（计算与语言） | `cs.AI`（人工智能） | `cs.HC`（人机交互） |

### 3.2 现有论文的证据

上述所有相关论文均发表在arXiv上，确认这是一个被认可的研究方向：
- NarrativeLoom: cs.HC, cs.MA
- StorySage: cs.HC, cs.AI, cs.MA
- ConStory-Bench: cs.CL, cs.AI

### 3.3 发表要求

要在arXiv上发表，需要补充以下内容：

| 要求 | 当前状态 | 需要行动 |
|------|----------|----------|
| 理论框架 | ✅ 完成 | 无 |
| 系统架构 | ✅ 完成 | 无 |
| 系统实现 | ⚠️ 部分完成 | 完成hybrid_rag_prototype |
| 用户研究 | ❌ 缺失 | 至少20-50名参与者 |
| 量化评估 | ❌ 缺失 | 定义指标，运行实验 |
| 基线对比 | ❌ 缺失 | 与NarrativeLoom、StorySage对比 |
| 开源代码 | ⚠️ 部分完成 | 完善并发布GitHub仓库 |

### 3.4 建议的评估指标

1. **质量指标**：
   - 叙事一致性（采用ConStory-Bench分类体系）
   - 用户满意度（李克特量表）
   - 专家评分（Torrance测试维度）

2. **效率指标**：
   - 完成故事所需时间
   - 需要的迭代次数
   - 用户干预频率

3. **领域专属指标**：
   - 爽点密度
   - 节奏评分
   - 与网文惯例的一致性
