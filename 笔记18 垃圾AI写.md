# 笔记18：L2编排层架构重构——从文件队列到对象传递

> 提出时间：2026-05-06
> 核心思想：将管道模式从"文件队列"升级为"对象传递"，专家从单次发言升级为Agent多轮迭代
> 本质简化：**对象是文件容器，专家处理时读取对象内的文件列表，产出追加到对象中**

---

## 核心洞察（2026-05-06补充）

**对象的本质就是文件容器**：
```
对象 = {
  对象名（如：卷纲规划_废土双界）,
  [初始文件.md],           // 用户输入，来自输入源节点
  [专家A意见.md, 聊天内容.json],  // 第一个专家的产出
  [专家B意见.md, 聊天内容.json],  // 第二个专家的产出
  [专家C意见.md, 聊天内容.json],  // 第三个专家的产出
  ...
}
```

**关键理解**：
- 专家节点从队列中读取的不是"文件"，而是"对象"
- 处理对象时，专家读取对象内的**所有已有文件**（初始文件+上游专家的全部产出）
- 专家完成处理后，将**自己的产出追加到对象中**
- 随着流程推进，对象内的文件列表**越积越多**
- 下游专家可以看到**完整的上游讨论历史**

---

## 一、现有架构的问题

### 1.1 当前管道模式的局限

```
当前实现：
文件1.md → 专家A → 专家B → 专家C → 输出
文件2.md → 专家A → 专家B → 专家C → 输出

问题：
- 每个专家只处理一次，产出是临时文本
- 专家之间通过简单字符串传递上下文
- 没有结构化的"意见沉淀"机制
- 下游专家无法看到上游的完整思考过程
```

### 1.2 专家角色的本质矛盾

当前的"专家"本质上是**Prompt模板**的实例化：
- 接收上下文 → 调用LLM一次 → 产出文本
- 没有多轮迭代能力
- 没有自我修正机制
- 产出是非结构化的Markdown文本

**矛盾**：真实的专家协作应该是持续迭代的过程，而非一次性发言。

## 二、新架构：对象传递模型

### 2.1 核心概念：PipelineObject（管道对象）

**极度简化的文件容器模型**：

```typescript
interface PipelineObject {
  // 对象标识
  id: string;
  name: string;           // 如："卷纲规划_废土双界"
  
  // 文件列表（按产生顺序排列）
  files: PipelineFile[];
  
  // 处理状态
  status: 'pending' | 'processing' | 'completed';
  currentNode: string;    // 当前正在处理的节点ID
  
  // 元数据
  metadata: {
    projectId: string;
    createdAt: string;
    tags: string[];
  };
}

interface PipelineFile {
  id: string;
  filename: string;       // 如："初始大纲.md"、"资深作者意见.md"
  content: string;        // 文件内容
  type: 'input' | 'report' | 'chat';  // 文件类型
  producer: {             // 产生者信息
    nodeId: string;
    expertType: string;
    expertId: string;
  };
  createdAt: string;
}
```

**对象在磁盘上的实际存储形式**：
```
data/projects/{projectId}/pipeline_objects/{objectId}/
├── manifest.json              # 对象元数据
├── 01_初始愿景.md            # 用户输入
├── 02_资深作者意见.md         # 专家A产出
├── 02_资深作者_聊天内容.json   # 专家A的迭代过程
├── 03_读者代表意见.md         # 专家B产出
├── 03_读者代表_聊天内容.json   # 专家B的迭代过程
└── ...
```
```

### 2.2 Agent专家的工作流程

**关键理解：专家读取的是对象内的"文件列表"**：

```
专家Agent处理流程：

1. 从队列中取出 PipelineObject
   ↓
2. 读取对象内的所有文件
   - 初始文件.md（用户输入）
   - 专家A意见.md + 聊天内容.json（上游产出）
   - 专家B意见.md + 聊天内容.json（上游产出）
   - ...
   ↓
3. Agent多轮内部讨论（生成→审视→修正）
   ↓
4. 产出两份文件：
   - {专家名}意见.md（最终报告）
   - {专家名}_聊天内容.json（迭代过程）
   ↓
5. 将这两份文件**追加到对象中**
   ↓
6. 将更新后的对象**推给下游节点**
```

**示意图**：
```
初始对象：
├── 01_初始大纲.md

↓ 资深作者节点处理

对象变为：
├── 01_初始大纲.md
├── 02_资深作者意见.md      ← 新增
└── 02_资深作者_聊天.json    ← 新增

↓ 读者代表节点处理（读取全部3个文件）

对象变为：
├── 01_初始大纲.md
├── 02_资深作者意见.md
├── 02_资深作者_聊天.json
├── 03_读者代表意见.md      ← 新增
└── 03_读者代表_聊天.json    ← 新增

↓ 剧情架构师节点处理（读取全部5个文件）

对象变为：
├── 01_初始大纲.md
├── 02_资深作者意见.md
├── 02_资深作者_聊天.json
├── 03_读者代表意见.md
├── 03_读者代表_聊天.json
├── 04_架构师意见.md        ← 新增
└── 04_架构师_聊天.json      ← 新增

...以此类推
```

**下游专家能看到什么**：
- 用户的原始输入
- 上游专家的最终意见
- 上游专家的完整思考过程（聊天记录）
- 因此可以**针对性地回应分歧、补充遗漏、修正错误**

### 2.3 多轮迭代的实现方式

**方式A：内部对话链（单Agent自迭代）**
```
专家Agent内部：
第1轮：产出初稿
第2轮：检查初稿问题 → 修正
第3轮：验证修正合理性 → 微调
...直到满意或达到最大轮数
```

**方式B：子会议模式（多Agent协作）**
```
复杂专家节点内部可展开子会议：
- 主分析师（提出框架）
- 质疑者（挑战漏洞）
- 修正者（提出改进）
- 决策者（收敛共识）
```

**方式C：人机协作迭代**
```
第1轮：AI产出初稿
→ 用户审核（通过/修改/驳回）
第2轮：AI基于反馈修正
→ 用户再次审核
...直到用户批准
```

## 三、架构优势分析

### 3.1 对比现有方案

| 维度 | 当前管道模式 | 新对象传递模式 |
|------|-------------|---------------|
| **数据单元** | 单个文件内容 | PipelineObject（文件容器） |
| **队列传递** | 文件队列 | 对象队列 |
| **专家输入** | 单个字符串 | 对象内的所有文件列表 |
| **专家产出** | 临时文本字符串 | 追加到对象的文件 |
| **思考可见性** | 下游只能看到最终输出 | 下游可看到上游完整聊天记录 |
| **迭代能力** | 无（单次处理） | 内置多轮迭代（存为json） |
| **可审计性** | 低 | 高（对象就是完整档案） |

### 3.2 解决的核心问题

1. **专家产出的结构化**
   - 从非结构化Markdown → 结构化JSON
   - 下游可以精准提取：建议、风险、评分等

2. **思考过程的可追溯**
   - 下游专家可以看到上游的完整迭代过程
   - 避免"为什么这样建议"的黑盒问题

3. **专家自我修正能力**
   - 不再是一次性发言
   - 可以内部多轮审视和优化

4. **复杂输入的处理**
   - 不再局限于单一文件
   - 可以打包多个相关文件（如愿景+大纲+参考案例）

## 四、关键技术实现

### 4.1 数据流改造

```typescript
// 原PipelineEngine
class PipelineEngine {
  async *processQueue(files: FileTask[]): AsyncGenerator<PipelineEvent>
}

// 新ObjectPipelineEngine
class ObjectPipelineEngine {
  async *processQueue(objects: WorkObject[]): AsyncGenerator<PipelineEvent> {
    for (const obj of objects) {
      // 按DAG拓扑排序处理节点
      for (const node of this.getTopologicalOrder()) {
        const agent = this.createAgent(node);
        
        // Agent处理对象，产出报告
        const report = await agent.process(obj);
        
        // 报告插入对象
        obj.expertReports.push(report);
        obj.currentNode = node.id;
        
        yield { type: 'report_generated', data: { objId: obj.id, report } };
      }
      
      obj.status = 'completed';
      yield { type: 'object_completed', data: obj };
    }
  }
}
```

### 4.2 Agent专家基类

```typescript
abstract class ExpertAgent {
  abstract expertId: string;
  abstract config: AgentConfig;
  
  // 核心处理逻辑
  async process(obj: WorkObject): Promise<ExpertReport> {
    // 1. 准备上下文
    const context = this.buildContext(obj);
    
    // 2. 多轮迭代生成报告
    const iterations = [];
    let currentDraft = await this.generateInitial(context);
    
    for (let round = 1; round <= this.config.maxRounds; round++) {
      const revision = await selfReflect(currentDraft, context);
      
      if (revision.needsChange) {
        iterations.push({
          round,
          thought: revision.thought,
          revision: revision.content,
          trigger: revision.trigger
        });
        currentDraft = revision.content;
      } else {
        break; // 自认可，停止迭代
      }
    }
    
    // 3. 结构化解析为Report
    return this.parseToReport(currentDraft, iterations);
  }
  
  // 自我审视（可Override）
  protected async selfReflect(draft: string, context: Context): Promise<Revision> {
    const prompt = `请审视以下专家意见，检查是否有逻辑漏洞、考虑不周或表述不清的地方：

${draft}

如有问题，请说明：
1. 问题是什么
2. 如何修正
3. 修正后的完整版本

如没有问题，请回复"无需修正"。`;

    const response = await llm.invoke(prompt);
    return this.parseRevision(response);
  }
}
```

## 七、关键技术实现（简化版）

### 6.1 对象队列引擎

```typescript
class ObjectPipelineEngine {
  private queue: AsyncQueue<PipelineObject>;
  private nodes: PipelineNode[];
  
  async *process(objects: PipelineObject[]): AsyncGenerator<PipelineEvent> {
    // 初始化队列
    for (const obj of objects) {
      await this.queue.enqueue(obj);
    }
    
    // 按DAG顺序处理每个对象
    for (const obj of objects) {
      for (const node of this.getTopologicalOrder()) {
        obj.currentNode = node.id;
        
        // 创建Agent
        const agent = new ExpertAgent({
          expertType: node.expertType,
          maxRounds: node.config.maxRounds || 3
        });
        
        // Agent读取对象内所有文件，产出新文件
        const newFiles = await agent.process(obj);
        
        // 追加到对象
        obj.files.push(...newFiles);
        
        yield {
          type: 'node_completed',
          data: { objectId: obj.id, nodeId: node.id, newFiles }
        };
      }
      
      obj.status = 'completed';
      yield { type: 'object_completed', data: obj };
    }
  }
}
```

### 6.2 ExpertAgent实现

```typescript
class ExpertAgent {
  constructor(private config: AgentConfig) {}
  
  async process(obj: PipelineObject): Promise<PipelineFile[]> {
    // 1. 读取对象内所有文件
    const contextFiles = obj.files;
    
    // 2. 多轮迭代（内部聊天）
    const chatHistory = [];
    let currentDraft = '';
    
    for (let round = 1; round <= this.config.maxRounds; round++) {
      const response = await llm.invoke(this.buildPrompt(contextFiles, currentDraft));
      
      chatHistory.push({
        round,
        role: 'assistant',
        content: response,
        timestamp: Date.now()
      });
      
      // 自审视
      const reflection = await this.selfReflect(response);
      if (!reflection.needsRevision) break;
      
      chatHistory.push({
        round,
        role: 'reflection',
        content: reflection.thought,
        revision: reflection.revision
      });
      
      currentDraft = reflection.revision;
    }
    
    // 3. 产出两份文件
    const reportFile: PipelineFile = {
      id: generateId(),
      filename: `${this.config.expertType}意见.md`,
      content: this.formatReport(currentDraft),
      type: 'report',
      producer: { nodeId: obj.currentNode, expertType: this.config.expertType, expertId: this.config.expertId },
      createdAt: new Date().toISOString()
    };
    
    const chatFile: PipelineFile = {
      id: generateId(),
      filename: `${this.config.expertType}_聊天内容.json`,
      content: JSON.stringify(chatHistory, null, 2),
      type: 'chat',
      producer: { nodeId: obj.currentNode, expertType: this.config.expertType, expertId: this.config.expertId },
      createdAt: new Date().toISOString()
    };
    
    return [reportFile, chatFile];
  }
  
  private buildPrompt(files: PipelineFile[], currentDraft: string): string {
    // 读取所有文件内容作为上下文
    const context = files.map(f => 
      `=== ${f.filename} ===\n${f.content}\n`
    ).join('\n');
    
    return `你是${this.config.expertType}，请基于以下材料发表意见：

${context}

${currentDraft ? `当前草稿：\n${currentDraft}\n请审视并修正：` : '请发表你的专业意见：'}`;
  }
}
```

### 9.3 技术实现

#### ZIP解析流程

```typescript
class DiscussionReplayLoader {
  async loadFromZip(zipFile: File): Promise<DiscussionSession> {
    const zip = await JSZip.loadAsync(zipFile);
    
    // 1. 读取config.cfg
    const configContent = await zip.file('config.cfg')?.async('string');
    const config = JSON.parse(configContent!);
    
    // 2. 遍历所有对象zip
    const objects: ReplayObject[] = [];
    
    for (const [filename, file] of Object.entries(zip.files)) {
      if (filename.endsWith('.zip') && !filename.startsWith('_')) {
        const objZipData = await file.async('blob');
        const objZip = await JSZip.loadAsync(objZipData);
        
        const obj = await this.parseObjectZip(objZip, filename);
        objects.push(obj);
      }
    }
    
    return {
      config,
      objects,
      exportTime: config.exportTime,
      totalStats: config.statistics
    };
  }
  
  private async parseObjectZip(objZip: JSZip, objName: string): Promise<ReplayObject> {
    const files: ReplayFile[] = [];
    
    // 按文件名排序（01_, 02_, 03_...）
    const sortedFiles = Object.values(objZip.files)
      .filter(f => !f.dir)
      .sort((a, b) => a.name.localeCompare(b.name));
    
    for (const file of sortedFiles) {
      const content = await file.async('string');
      const type = this.detectFileType(file.name);
      
      files.push({
        name: file.name,
        type,
        content,
        parsedContent: type === 'chat' ? JSON.parse(content) : content
      });
    }
    
    return {
      name: objName.replace('.zip', ''),
      files,
      steps: this.buildTimeline(files)
    };
  }
  
  private detectFileType(filename: string): 'input' | 'report' | 'chat' {
    if (filename.includes('聊天') || filename.endsWith('.json')) return 'chat';
    if (filename.startsWith('01_')) return 'input';
    return 'report';
  }
  
  private buildTimeline(files: ReplayFile[]): TimelineStep[] {
    const steps: TimelineStep[] = [];
    let currentStep: TimelineStep | null = null;
    
    for (const file of files) {
      if (file.type === 'input') {
        steps.push({
          step: 0,
          type: 'input',
          title: '初始输入',
          files: [file],
          timestamp: null
        });
      } else if (file.type === 'report') {
        // 从文件名提取专家类型：如"资深作者意见.md"
        const expertType = file.name.match(/\d+_(.+?)意见\.md/)?.[1] || '未知专家';
        
        currentStep = {
          step: steps.length,
          type: 'expert',
          title: expertType,
          expertName: expertType,
          reportFile: file,
          chatFile: null,
          timestamp: this.extractTimestamp(file.content)
        };
        steps.push(currentStep);
      } else if (file.type === 'chat' && currentStep) {
        currentStep.chatFile = file;
      }
    }
    
    return steps;
  }
}

## 十、结论

这个新架构的本质是：
> **对象是文件容器，在专家间传递；专家读取对象内的所有文件，产出追加到对象中**

它解决了当前系统的核心痛点：
1. **专家产出的结构化**：意见.md + 聊天.json，而非临时文本
2. **思考过程可见**：下游专家可以看到上游的完整迭代过程
3. **自我修正能力**：Agent内部多轮讨论，产出质量更高
4. **灵活配置**：未来可支持选择性读取，控制上下文长度

这是一个向**智能体协作网络**演进的关键步骤，同时保持了实现的简洁性（本质就是文件操作）。
