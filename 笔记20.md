# 笔记20：世界书（World Book）深化设计

## 一、最终实现状态（2026-05-10）

世界书是关键词触发的知识注入系统。参照 SillyTavern World Info 架构，实现了多书存储、关键词过滤注入、PNG 角色卡导入。

**v5.0 新增（同日）**：世界书管理员——自动从专家/群聊讨论中提取事实并更新条目。画布上可放置观察者图标节点，专家/群聊节点启用"世界书总结"开关后，聊天内容自动推入管理员队列，LLM 分析产出 create/update/merge/skip 动作，支持自动/半自动/手动三种模式。

### 1.1 数据模型（WorldBookEntry 完整字段）

| 字段 | 类型 | 默认值 | 状态 | 说明 |
|------|------|--------|------|------|
| `id` | string | — | ✓ | 唯一标识 |

| `secondaryKeys` | string[] | [] | ✓ | 次要触发词 |

| `selective` | boolean | true | ✓ | 启用选择性逻辑 |
| `selectiveLogic` | enum | AND_ANY | ✓ | AND_ANY / NOT_ALL / NOT_ANY / AND_ALL |
| `priority` | number | 10 | ✓ | 排序权重 |
| `position` | string | before_char | ✓ | 注入位置（before_char/after_char/an_top/an_bottom/at_depth/em_top/em_bottom） |
| `disable` | boolean | false | ✓ | 禁用条目 |

### 1.4 注入架构（关键实现）

```
Pipeline 启动
    │

        └── ExpertContext.worldbook = 过滤后的文本
```

## 二、世界书管理员（World Book Manager）设计

### 2.1 问题

当前世界书只实现了"注入"方向：pipeline 启动时加载条目，Worker 运行时按关键词过滤注入到专家上下文。但世界书的条目从哪来？目前只能靠用户手动在 WorldBook.vue 页面上逐条填写。这在长篇创作中是不可持续的——随着故事推进，新角色登场、主角突破境界、世界观设定展开，世界书应该持续**自动增长**。

之前笔记7-9中讨论过世界书管理 Agent 的概念，但由于当时重心在注入链路，更新功能一直没有落地。

### 2.2 核心设计：世界书管理员节点

**定位**：世界书管理员是一个**观察者**而非**参与者**。它不处在数据流的必经路径上，不接收 PipelineObject，不产出报告。它监听专家/群聊节点的产出，异步提取信息并写入世界书。

**与专家/群聊节点的关系**：

```
专家节点 A ──数据流──→ 专家节点 B ──数据流──→ 输出节点
    │                      │
    │  世界书总结: ✓       │  世界书总结: ✗
    │  (触发管理员)        │  (不触发)
    │                      │
    └──────────┬───────────┘
               │  聊天内容
               ▼
        世界书管理员
        (观察者，非数据流节点)
               │
               ▼
        worldbooks/*.json
```

**关键约束**：
- 世界书图标在画布上是**配置型节点**（类似设置面板），不与 ExpertNode/GroupNode 连线
- 双击世界书图标打开管理员窗体，可实时查看其工作状态和消息流
- 管理员有自己的 LLM 调用，使用轻量模型（temperature 0.3），输出结构化 JSON
- 只由专家/群聊节点**显式启用**才触发，默认关闭

### 2.3 触发条件

每个专家节点和群聊容器节点新增一个开关 `enableWorldBookSummary`（默认 false）。

触发时机：当启用此开关的节点完成以下任一事件后，将该节点的聊天内容推入世界书管理员的处理队列：

| 触发事件 | 推送内容 |
|---------|---------|
| Agent 轮次完成（`agent_round_complete`） | 该轮 assistant 产出内容 |
| Agent 完成（`agent_complete`） | 完整 chatLog（所有轮次） |
| 群聊轮次完成（`group_chat_round_complete`） | 该轮所有成员发言内容 |
| 群聊完成（`group_chat_complete`） | 完整群聊 chatLog |

对于高频场景（每轮都触发），管理员内部做**去重和合并**：同一对象的相邻推送如果涉及同一主题，合并为一次处理。

### 2.4 世界书管理员的处理流程

```
聊天内容进入队列
      │
      ▼
  ┌─────────────────────────────┐
  │ 世界书管理员 LLM 调用       │
  │                              │
  │ Prompt 包含：                │
  │ - 当前世界书已有条目列表     │
  │ - 本次聊天内容               │
  │ - 指示：提取可入库的信息     │
  │                              │
  │ 输出 JSON 格式：             │
  │ {                            │
  │   "actions": [               │
  │     {                        │
  │       "action": "create"|    │
  │                "update"|     │
  │                "merge"|      │
  │                "skip",       │
  │       "keys": ["关键词1"],   │
  │       "content": "条目内容", │
  │       "targetEntryId": "..", │
  │       "reason": "原因"       │
  │     }                        │
  │   ]                          │
  │ }                            │
  └──────────┬──────────────────┘
             │
             ▼
  ┌─────────────────────────────┐
  │ 执行动作（自动模式）        │
  │ 或等待用户审核（半自动模式）│
  └──────────┬──────────────────┘
             │
             ▼
  worldbooks/<bookId>.json 更新
```

**四种动作**：
- **create**：新信息，创建条目
- **update**：信息已有但需要补充修改（如"主角从金丹期突破到元婴期"）
- **merge**：两条条目描述同一事物，建议合并
- **skip**：聊天内容无新信息，跳过

**去重策略**：管理员在决定 create 前检查现有条目的 keys 和 content 相似度（简单的关键词重叠率），避免创建重复条目。

**三栏布局**：
- **左栏 — 监控节点**：列出画布上所有启用了世界书总结的节点，带开关状态指示灯（绿=工作中，灰=未触发）
- **中栏 — 处理日志**：按时间倒序展示每次处理请求的 LLM 交互——输入摘要、思考过程（可折叠）、输出动作
- **右栏 — 建议条目预览**：待用户确认的条目修改建议，支持逐条确认/修改/拒绝；已处理的历史记录

**操作模式**（窗体顶部下拉切换）：
- **自动模式**（auto）：管理员直接写入世界书，左下角 toast 通知用户
- **半自动模式**（semi_auto，默认）：管理员暂停在"建议条目预览"，等用户确认
- **手动模式**（manual）：管理员只分析不执行，所有建议排队等待用户手动操作

### 2.7 后端架构

#### 新增文件

```
backend-node/src/services/
  worldbook-manager.ts    # WorldBookManager Agent 核心逻辑
  worldbook-summarizer.ts # LLM 总结 Prompt 构建 + JSON 解析
```

#### 新增 Worker

在 `pipeline-workers.ts` 中新增 `worldbookObserverWorker`（与六种现有 Worker 并列）：

- **输入**：不从对象队列取，而是监听一个专门的 `worldbookTaskQueue: AsyncQueue<WorldBookSummaryTask>`
- **触发**：expertWorker / groupChatWorker 在节点完成后检查 `enableWorldBookSummary`，如果启用则 `enqueue(task)` 到该队列
- **执行**：调用 `WorldBookSummarizer.summarize(task)` → 返回 actions → `WorldBookExecutor.execute(actions)`
- **SSE 事件**：独立事件类型 `worldbook_*`，前端世界书管理员窗体通过 BroadcastChannel 监听

#### 新增 SSE 事件

```
worldbook_task_queued     → 任务已排队
worldbook_analyze_start   → 开始分析聊天内容
worldbook_analyze_chunk   → 管理员 LLM 流式输出（thinking/content）
worldbook_actions_ready   → 分析完成，返回建议 actions
worldbook_action_applied  → 单条 action 已执行
worldbook_task_complete   → 任务处理完成
```

#### 新增 API

```
POST /api/projects/:id/worldbooks/:bookId/summarize
  body: { chatContent, nodeId, objectId, mode: 'auto'|'semi_auto'|'manual' }
  → 单次总结请求（手动触发）

GET  /api/projects/:id/worldbooks/manager/stream?meetingId=xxx
  → SSE 流，订阅世界书管理员事件

PUT  /api/projects/:id/worldbooks/:bookId/actions/:actionId
  → 确认/修改/拒绝单条 action（半自动模式）
```

### 2.9 管理员与现有 WorldBookManager 的关系

现有 `services/worldbook.ts` 中的 `WorldBookManager` 类是一个占位骨架（`processSequence` 返回空数组）。新设计的世界书管理员是它的**完整实现**：

- 废弃 `processSequence()` → 替换为 `processChatContent(chatContent, existingEntries)` 调用 LLM
- 保留 `getConflicts()` 用于冲突检测
- 新增 `applyActions(actions, worldbook)` 用于执行写操作

#### SSE 事件（通过 BroadcastChannel 广播）

| 事件类型 | 发射时机 | 关键字段 |
|---------|---------|---------|
| `worldbook_task_queued` | task 入队后 | taskId, nodeId, nodeType, objectId, triggeredBy |
| `worldbook_analyze_start` | LLM 调用前 | taskId, nodeId, chatContentLength, bookId |
| `worldbook_actions_ready` | LLM 解析完成 | taskId, actions[], summary, createCount/updateCount/mergeCount/skipCount |
| `worldbook_action_applied` | 单条 action 写入后 | taskId, actionId, actionType, keys, bookId |

#### 触发数据流

```
专家/群聊节点(enableWorldBookSummary=true)
    │
    ├── per_round: agent_round_complete / group_chat_round_complete 后触发
    └── per_node: agent_complete / group_chat_complete 后触发
           │
           ▼
    tryEnqueueWorldBookTask(nodeId, obj, state, ...)
           │  检查 nodeConfig.worldbookSummaryConfig
           │  按 triggerGranularity 过滤
           │  构建 WorldBookSummaryTask
           │
           ▼
    state.worldbookTaskQueue.enqueue(task)
           │
           ▼
    worldbookObserverWorker (独立循环, sleep 1s 轮询)
           │  dequeueSync() → processTask()
           ▼
    WorldBookManagerService
           │  buildSummarizerPrompt(task)
           │  LLM.invoke(prompt, {temperature:0.3, maxTokens:4096})
           │  parseAnalysisResponse(raw)
           │
           ├── auto: applyAction() 直接写入 → worldbook_task_complete
           └── semi_auto: actions 存入 pendingActions Map
                         → worldbook_actions_ready 广播
                         → 前端管理员窗确认 → PUT API → applyAction()
```
