# 笔记12：L1.5——情节编排层

## 一、L1-L4模型的理想化问题

### 1.1 开发中暴露的裂缝

经过实际开发，L1-L4四层模型暴露出一个显著的逻辑跳跃：

```
L1 种子层
  输出：故事愿景（核心梗、人设、粗略大纲、金手指、风格）
        ↓
        这里是断裂带
        ↓
L2 架构层
  输入：L1 愿景 + 世界书 + RAG
  输出：精修大纲（功能→序列→情节，带情绪标注）
```

L1 给出的粗略大纲通常是几百字的叙事散文，而 L2 需要的是结构化的功能序列。从散文到结构化的跳跃太大，L2 三专家讨论时经常需要反复追问"这一段到底包含几个序列？范围多大？"

### 1.2 软件工程的类比

软件开发不会从需求文档直接跳到函数设计。中间必定有系统设计层——处理模块之间的关系、决定架构、划分边界。L1.5 就是这个缺失层。

---

## 二、L1.5 的定位

### 2.1 在瀑布模型中的位置

```
L1   种子层       故事愿景      "我要写一个什么样的故事"
L1.5 情节编排层   卷纲          "这个故事分成几卷，每卷走向如何"
L2   架构层       章纲          "第30-40章每一章的精确剧情"
L3   叙事层       带标签草稿    "根据章纲写出粗略草稿，嵌入叙事指令"
L4   渲染层       精修正文      "根据标签指令扩展打磨成终稿"
```

L1.5 输出的是**卷纲**——以卷为单位，讨论这一卷的方向、热点、读者预期，给出每章的简要方向。L2 接收卷纲后，就指定章节范围进行**章纲**级别的深入讨论。

> **注**：L3 和 L4 的重新定位（带标签草稿 + 多技能体系）见[笔记13](笔记13.md)。

### 2.2 两专家会议：资深作者 + 读者

L1.5 也开专家会议，但侧重与 L2 完全不同——这里不需要逻辑推演和 OOC 检测，需要的是**方向决策**。

| 专家 | 角色 | 核心职责 |
|------|------|---------|
| **资深作者** | 创作方向把控 | 热点判断、爽点方向、毒点预警、商业趋势、结构经验 |
| **读者代表** | 体验审核者 | 读者预期管理、疲劳度检测、代入感评估 |

> 资深作者之前暂定为网络编辑。改名原因：L2 也有网编，同名易混淆。两者 prompt 本质相同，但角色感不同——一个是经验丰富的老作者和你讨论方向，一个是编辑审核稿子。

**典型讨论议题**：

- "这一卷该不该给主角安排强敌？没对手太顺，太强又压节奏"
- "该不该开新地图？新鲜感 vs 稀释现有戏份"
- "这个配角该不该推下去？留着占戏份但读者喜欢"
- "下面的剧情往哪个方向展开？三个可能方向各有什么利弊"

这些讨论的产物是**方向性决策**，不是功能序列。L2 拿到的是"已经确定方向"的章节概要，直接拆功能序列。

### 2.3 与 L2 三专家会议的对比

| 维度 | L1.5 两专家 | L2 三专家 |
|------|-----------|----------|
| **参与者** | 资深作者 + 读者 | 剧情架构师 + 网编 + 人物设计师 |
| **产出** | 卷纲（每章方向性描述） | 章纲（功能序列 + 情绪曲线） |
| **粒度** | 章级方向（1-3句/章） | 序列/功能级（表格化） |
| **讨论焦点** | 方向对不对？读者买不买账？ | 逻辑通不通？功能链完整？人物OOC？ |
| **资深作者/网编** | 主导——决定方向 | 评议——审核爽点执行 |

L1.5 不做逻辑推导。讨论"该不该开新地图"时不需要架构师拆序列，需要的是资深作者判断市场趋势，读者判断会不会疲劳。方向定好后，L2 接手落地。

## 三、输入与输出

### 3.1 输入

| 来源 | 内容 |
|------|------|
| L1 愿景文档 | 核心梗、人设、金手指、粗略大纲、风格基调、目标篇幅 |
| 用户补充 | 是否多主角？多时间线？是否有想好的卷名/高潮？ |
| 世界书 | 已有的世界观设定（L1 阶段已填写的核心层） |

### 3.2 输出

一组 `.md` 文件，放在文档库中：

### 3.3 输出粒度

| 项 | 粒度 |
|----|------|
| 每章概括 | 1-3 句话，不超过 100 字 |
| 每卷篇幅 | 10-50 章 |
| 一次生成量 | 至少覆盖 L2 下一轮要讨论的章节数（如 5-10 章） |
| 衔接说明 | 2-3 句，说明前因后果和情绪过渡 |

## 四、L1.5 两专家会议的工作过程

### 4.1 会议结构

只有一种驱动模式（不需要 L2 的三种），因为 L1.5 天然是市场导向：

```
发言顺序：
  1. 资深作者（主导） → 提出分卷方案、每卷方向建议
  2. 读者代表（审核） → 从读者视角评估，提出质疑
  3. 资深作者根据读者反馈调整
  4. 用户（主编）决策
  5. 如未达成共识，循环
```

### 4.3 资深作者 vs L2 网编

同一套市场判断力，不同发言层级：

| | L1.5 资深作者 | L2 网编 |
|--|----------|--------|
| **粒度** | 卷级方向、读者趋势 | 章级爽点执行、情绪曲线 |
| **发言** | 主导——提方向 | 评议——审执行 |
| **例子** | "这卷应该安排一个强敌" | "第32章打脸铺垫太短" |

实现时同一个 `WebEditorV1` 类，通过 `context.layer` 切换 prompt 前缀：

```python
class WebEditorV1(BaseExpert):
    def speak(self, outline, context):
        layer = context.get("layer", "l2")
        
        if layer == "l1_5":
            prompt = L1_5_AUTHOR_PROMPT    # 资深作者：卷级方向讨论
        else:
            prompt = L2_EDITOR_PROMPT      # 网编：章级爽点审核
        
        return self._invoke_with_prompt(prompt, outline, context)
```

### 4.5 上下文注意力管理

本卷讨论内容放在上下文注意力最强的区域（prompt 末尾），背景资料（L1 愿景、世界书核心层）放中低注意力区域。分层注入，避免当前焦点被长背景淹没。

## 五、增量规划

### 5.1 最小可行输出

每次至少覆盖 L2 下一轮要讨论的范围即可。先写 5 章试试水 → L2 讨论 → 满意再回来继续。

### 5.2 断点续做

文档库中的草稿状态：

```
📂 L1.5_规划
  📄 卷一_宗门崛起.md     ← completed
  📄 卷二_历练篇.md       ← draft（只规划到第40章）
  📄 卷三_争霸篇.md       ← pending
```

打开卷二，系统读取前 40 章概要作为上下文，从第 41 章继续。

### 5.3 L1.5 ↔ L2 反馈循环

L2 讨论中发现方向问题 → 两种处理方式：

- **大问题**（方向性错误、章节顺序需要重排）：声明原因，打回 L1.5 重新编排
- **小问题**（单章细节不合理、衔接生硬）：L2 自行在当前范围内调整，不触发 L1.5 重新规划

### 5.4 L1.5 的 RAG 策略

不需要。L1.5 讨论的内容（一个卷的章节概要）全塞进上下文最多几十 KB，用不到 RAG 检索。只需要挂世界书——查询当前的人物状态、已有伏笔、力量体系规则，确保分卷方案不撞设定。

## 六、与 L1 和 L2 的连接

### 6.1 L1 → L1.5

L1 完成后弹窗建议进入 L1.5（长篇/多线时），也可跳过直接进 L2（短篇/单线）。

L1 本身是作者的创意发挥空间，不应有太多约束。但至少需要一个专门的 skill：**归纳整理**——把作者在 L1 聊天中散落的创意讨论（题材、人设、金手指、世界观碎片）归纳成一份结构化的愿景草稿，方便 L1.5 和 L2 作为输入。L3/L4 的多技能体系（[笔记13](笔记13.md)）不适用于 L1——L1 就是简单 prompt 引导创作冲动。

### 6.2 L1.5 → L2：卷纲 → 章纲

L2 接收 L1.5 的 `.md`，在指定章节范围内展开章纲级别的功能序列。

```
L1.5 产出：卷二 历练篇.md
  ├── 第31-35章：主角个人线（方向已定：秘境入口探索）
  ├── 第36-40章：李寒视角线（方向已定：身世秘密揭晓）
  └── ...

        ↓ L2 接收，对第31-40章展开

L2 产出：第31-40章 章纲（功能序列级别）
  ├── 第31章
  │   ├── 功能链：铺垫(秘境传闻) → 获得(发现入口) → 违背(无视禁令)
  │   ├── 情绪曲线：好奇→兴奋→紧张
  │   ├── 人物：主角(主体)、守门弟子(阻碍者)
  │   └── 详细剧情：...
  ├── 第32章
  │   └── ...
  └── ...
```

L1.5 决定"第31-35章写秘境探索"，L2 决定"具体功能链怎么排来实现这个探索"。

## 八、会议协议

```python
class EditorReaderDriven(BaseMeetingProtocol):
    """L1.5 两专家会议：资深作者主导 + 读者审核"""
    
    def get_speaking_order(self, context):
        return [
            ("author", "main"),
            ("reader", "review"),
            ("author", "supplement"),
        ]
    
    def should_continue(self, rounds, consensus):
        return rounds < 3
```

只有一种驱动模式，不需要 L2 的三种。

### 读者代表的约束

读者代表是 L1.5 特有的角色，不属于 L2。职责：
- **模拟读者情绪**："看到这里我会觉得无聊/爽/困惑/没耐心"
- **检测疲劳**："连续 10 章都是宗门内场景，读者会疲劳"
- **质疑方向**："为什么主角不去探索秘境？读者会觉得不合理"
- **预测反应**："如果第 40 章杀了一个人气配角，读者可能会骂"
- **不负责解决方案**：只提"哪里可能有问题"，方案由资深作者和用户讨论

实现时应有 `READER_PROMPT` 明确约束：
- 严禁提出"应该怎么写"——这是资深作者和架构师的事
- 只表达"读者会怎么感受"
- 必须以读者口吻发言，如"我看到第XX章的时候会觉得……"

## 九、层间关系总结

```
L1 (种子) ──→ L1.5 (卷纲) ──→ L2 (章纲) ──→ L3 (带标签草稿) ──→ L4 (正文)
   │             │              │              │                    │
   │         两专家会议      三专家会议      标签嵌入草稿         精修润色
   │      资深作者+读者   架构+网编+人物    (见笔记13)         (见笔记13)
   │             │              │
   │         方向决策        执行拆解
   │       "写什么方向"   "怎么写出来"
   │         市场优先        逻辑优先
```
