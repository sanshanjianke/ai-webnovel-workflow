# 笔记16：L1.5/L2 合并——画布编排下的统一专家会议

## 一、L1.5 和 L2 分离的问题

### 1.1 本质相同

观察当前代码：

```
L1.5 API (backend/api/l1_5.py):
  ConfigurableExpertMeeting(config)
    → for round in max_rounds:
        for expert in config.experts:
          expert.speak() → yield SSE

L2 API (backend/api/l2.py):
  ExpertMeetingL2(protocol_id, max_rounds)
    → protocol.get_speaking_order()
    → for expert_type in order:
        get_module("expert", expert_map[expert_type])
        expert.speak() → yield SSE
```

两者都是：取专家队列 → 按序发言 → SSE流式输出 → 格式化产出。

区别只在：
| 维度 | L1.5 | L2 |
|------|------|-----|
| 专家组合 | 资深作者+读者 | 架构师+网编+人物设计师 |
| 默认粒度 | 卷级 | 章级 |
| 注册位置 | `modules/l1_5/` | `modules/l2/experts/` |

这些差异完全通过配置解决，不需要两个独立层级。

### 1.2 当前代码的冗余

- `modules/l1_5/__init__.py`：5个可配置专家（SeniorAuthorV1、ReaderRepresentativeV1、PlotArchitectV1、CharacterDesignerV1、WebEditorV1）
- `modules/l2/experts/`：3个独立专家（architect.py、character.py、editor.py）
- 两组专家功能重叠（PlotArchitectV1 vs architect.py 中的 PlotArchitect）
- L1.5 的 API 已实现 `ConfigurableExpertMeeting`，比 L2 的 `ExpertMeetingL2` 更灵活
- L2 的 ExpertMeetingL2 用硬编码 `expert_map` 映射专家类型

### 1.3 固定层级对用户的限制

```
当前：L1 → [L1.5 → L2 → L3 → L4]
                          
问题1：短篇不需要分卷，L1.5多余
问题2：长篇可能想 L1→L2→L1→L2 迭代
问题3：用户可能只想用网编快速审核一份大纲
问题4：新增专家必须修改层级结构
```

## 二、新架构：统一编排层

### 2.1 层级模型

```
L1 种子层        → 故事愿景（不变）
                     ↓
编排层            → 可自由组合的专家会议
 ┌─────────────────────────────┐
 │  画布：拖拽专家节点，连线排序 │
 │                             │
 │  预设模板：                  │
 │  ├─ 快速审核（网编）         │
 │  ├─ 卷纲规划（作者+读者）    │
 │  ├─ 章纲设计（架构+编辑+人物）│
 │  └─ 自定义...               │
 │                             │
 │  专家池：                    │
 │  ├─ 资深作者（方向/市场）    │
 │  ├─ 读者代表（体验审核）     │
 │  ├─ 剧情架构师（逻辑/功能）  │
 │  ├─ 人物设计师（OOC/行动元） │
 │  ├─ 网络编辑（爽点/毒点）    │
 │  └─ 用户自定义skill...       │
 └─────────────────────────────┘
                     ↓
L3 叙事层        → 带标签草稿
L4 渲染层        → 精修正文
```

核心变化：
- **L1.5 和 L2 不再作为独立层级存在**
- 统一为"编排层"，通过画布自由组合专家
- 文本输出粒度（卷纲/章纲/场景）由用户选定，不绑定层级

### 2.2 节点模型

每个专家是一个画布节点：

```json
{
  "id": "node_1",
  "type": "expert",
  "expert_id": "senior_author_v1",
  "config": {
    "role": "main",
    "granularity": "volume",
    "custom_prompt": null
  },
  "position": { "x": 100, "y": 200 }
}
```

连线定义发言顺序：

```json
{
  "id": "edge_1",
  "source": "node_1",
  "target": "node_2",
  "type": "speak_order"
}
```

### 2.3 执行引擎

不再区分 L1.5 和 L2 的 API，统一入口：

```
POST /api/projects/{id}/orchestrate/start
  {
    "meeting_name": "卷纲编排",
    "granularity": "volume",
    "nodes": [
      {"expert_id": "senior_author_v1", "role": "main"},
      {"expert_id": "reader_representative_v1", "role": "review"},
      {"expert_id": "senior_author_v1", "role": "supplement"}
    ],
    "collaboration_mode": "semi_auto",
    "max_rounds": 3
  }
```

引擎就是当前的 `ConfigurableExpertMeeting`——它已经支持动态专家队列，只需要：
1. 从 `modules/l1_5/` 迁移到通用位置（如 `modules/orchestration/`）
2. 合并 L2 的旧 expert 实现到统一的专家池
3. 删除硬编码的 `ExpertMeetingL2`

## 四、API 合并方案

### 4.1 旧端点 → 新端点

| 旧 | 新 | 说明 |
|----|----|------|
| `POST /projects/{id}/l1_5/start` | `POST /projects/{id}/meeting/start` | 统一入口 |
| `GET /projects/{id}/l1_5/stream` | `GET /projects/{id}/meeting/stream` | 兼容预设 |
| `POST /projects/{id}/l1_5/feedback` | `POST /projects/{id}/meeting/feedback` | 用户反馈 |
| `GET /projects/{id}/l2/stream` | 废弃（重定向到 /meeting/stream?preset=chapter_design） | |
| `POST /projects/{id}/l2/message` | `POST /projects/{id}/meeting/feedback` | |
| `POST /projects/{id}/l2/decide` | `POST /projects/{id}/meeting/feedback` | |

### 4.2 兼容策略

L1.5 和 L2 的旧端点保留，内部重定向到新引擎。前端迁移期间，两个版本并存。

## 五、画布前端

### 5.1 实现方案

使用 Vue Flow（`@vue-flow/core`）实现拖拽编排画布：

```
┌───────────────────────────────────────────────┐
│  编排画布                     [保存] [运行]     │
│  ┌──────────────────────────────────────────┐ │
│  │                                          │ │
│  │  ┌──────────┐    ┌──────────┐            │ │
│  │  │ 资深作者  │───→│ 读者代表  │            │ │
│  │  │  role:主  │    │ role:审  │            │ │
│  │  └──────────┘    └──────────┘            │ │
│  │       │               │                  │ │
│  │       │    ┌──────────┘                  │ │
│  │       ▼    ▼                             │ │
│  │  ┌──────────┐                            │ │
│  │  │ 资深作者  │                            │ │
│  │  │ role:补充 │                            │ │
│  │  └──────────┘                            │ │
│  │                                          │ │
│  └──────────────────────────────────────────┘ │
│  ┌─────┐ ┌─────┐ ┌─────┐ ┌─────┐ ┌──────┐  │
│  │作者 │ │读者 │ │架构 │ │人物 │ │+自定义│  │
│  └─────┘ └─────┘ └─────┘ └─────┘ └──────┘  │
└───────────────────────────────────────────────┘
```

### 5.2 交互

- 从底部工具箱拖专家到画布
- 连线定义发言顺序
- 右键节点 → 配置面板：角色(main/review/supplement)、粒度、自定义prompt
- 画布配置保存为 JSON，下次打开恢复
- 点击 [运行] → 启动 SSE 会议流

### 5.3 预设模板

前端默认展示3个预设模板卡片，点击一键加载：

| 模板 | 专家队列 | 适用 |
|------|---------|------|
| 快速审核 | 网编(main) 1轮 | 短篇/快速检查 |
| 卷纲规划 | 作者(main) → 读者(review) → 作者(supp) 3轮 | 长篇分卷 |
| 章纲设计 | 架构(main) → 编辑(review) → 人物(supp) 3轮 | 逐章精修 |

## 六、实施路径

### 第一阶段：后端统一（不破坏现有API）

1. 创建 `backend/modules/experts/`，合并所有专家实现
2. 将 `ConfigurableExpertMeeting` 迁移到 `backend/modules/orchestration/`
3. 删除 `ExpertMeetingL2` 和旧的 l2/experts/ 目录
4. 更新 registry.py 的 `l1_5`/`l2` 分类为统一的 `expert`
5. 新增 `POST /api/projects/{id}/meeting/start` 统一端点
6. 旧端点内部转发到新引擎

### 第二阶段：前端画布

1. 安装 `@vue-flow/core`
2. 实现 `OrchestrationCanvas.vue`
3. 组合进 L1.5 和 L2 页面（共用一个画布组件）
4. 预设模板卡片 + 自定义模式

### 第三阶段：前端导航合并

1. 导航栏 L1.5 和 L2 合并为一个"编排"按钮
2. 旧页面保留作为兼容

## 七、与笔记14的关系

笔记14 提出的工作流画布是**系统级**的（L1→编排→L3→L4，所有模块可连线）。

本笔记的编排画布是**模块级**的（在编排层内部，专家节点可拖拽）。

两者层级不同：
- 模块级画布：本笔记实现，专家节点拖拽
- 系统级画布：笔记14长期愿景，模块间连线
