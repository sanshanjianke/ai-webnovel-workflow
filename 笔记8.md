# 笔记8：世界书——持久化状态管理与RAG的协同

## 一、对RAG的再审视

### RAG的根本局限

RAG返回的是**文本碎片**，不是**结构化认知**。向量相似度检索能告诉你"这段文本和你的查询相近"，但不能告诉你：

- 这个世界的力量体系规则是什么
- 主角当前的修为等级和人际关系
- 哪些伏笔已经埋下但尚未回收
- 当前剧情处于情绪曲线的哪个位置

笔记7的混合检索方案（多视角改写、多维度切片、向量+关键词混合）虽然改善了检索准确率，但本质上仍是**碎片检索**。RAG解决的是"找"的问题，没有解决"理解"的问题。

### 上下文窗口变大的影响

大模型正向1M+上下文发展，这意味着：

- **以前**RAG是为了突破上下文硬限制——塞不进去，不得不检索
- **现在**RAG的意义更多在于：
  1. **节省成本**：token费用仍然是实际约束
  2. **管理注意力**：lost-in-middle问题不会因窗口变大而消失，精选少量相关片段优于全量塞入
  3. **降低噪音**：让模型专注于当前任务，不被无关信息分散

**结论**：RAG仍然重要，但需要一个**结构化的状态管理系统**与之配合——这就是"世界书"。

## 二、世界书的概念

### 灵感来源：SillyTavern World Book

SillyTavern的World Book（又称WorldInfo/Lorebook）是一个基于**触发词**的上下文注入系统：

- 每个条目有一个或多个触发词（keys）
- 当对话中出现触发词时，对应条目被激活并注入上下文
- 条目可设优先级、递归深度、注入位置（在角色卡之前/之后/作者笔记中等）
- 支持"常驻条目"——始终存在于上下文中

核心设计思想：**不是"检索相关内容"，而是"维护世界状态"**。

### RAG的两种职能

RAG在我们的系统中实际上承担两种不同的角色，需要区分开来：

| 职能 | RAG-历史回顾 | RAG-技法参考 |
|------|-------------|-------------|
| 内容 | 前文章节片段、已发生剧情的摘要、历史对话 | 爽点公式、映射规则、参考文本范例 |
| 触发时机 | 创作过程中持续查询 | L3/L4执行时定向查询 |
| 数据来源 | 当前作品的已生成内容（不断增长） | 预先构建的静态知识库 |
| 解决的问题 | "之前发生了什么" | "怎么写更好" |
| 更新频率 | 每完成一个序列即索引 | 很少更新 |

**为什么历史回顾必须用RAG而非全塞入上下文？**

当小说超过百万字时：
1. **成本不可接受**：即使1M上下文窗口，每轮创作把200章全文塞进去，token费用极高
2. **注意力衰减**：lost-in-middle问题意味着模型对prompt中段的注意力显著低于首尾，全量塞入反而降低质量
3. **噪音干扰**：第300章写战斗场景时，第5章的日常描写只会分散注意力

历史回顾RAG的核心价值：**精准提取与当前创作相关的前文片段**，让模型在有限的注意力预算内获得最相关的历史信息。

### RAG vs 世界书

| 维度 | RAG（历史回顾） | RAG（技法参考） | 世界书 |
|------|----------------|----------------|--------|
| 激活方式 | 语义相似度 | 语义+标签 | 触发词+确定性 |
| 内容形态 | 前文文本片段/摘要 | 技法规则+范例 | 结构化状态条目 |
| 知识性质 | "之前发生了什么" | "怎么写更好" | "现在是什么状态" |
| 数据来源 | 当前作品（动态增长） | 静态知识库 | 随创作过程动态更新 |
| 解决的问题 | 历史把控、伏笔呼应 | 技法实现、风格一致 | 状态一致性 |

**核心差异**：
- RAG历史回顾 = **"回忆过去"**——第50章老角色第300章再出场，AI需要想起他
- 世界书 = **"把握当下"**——这个角色现在的状态、关系、修为是什么
- RAG技法参考 = **"怎么写"**——如何把这段写得好看

## 三、世界书在四层模型中的定位

世界书不是一个单独的层，而是一个**跨层的持久化状态层**，与RAG形成三元协同：

```
                        世界书 (World Book)
                        当前状态管理
                              │
    ┌─────────┬─────────┬─────┼─────┬─────────┬─────────┐
    │         │         │     │     │         │         │
    ↓         ↓         ↓     ↓     ↓         ↓         ↓
  世界观    人物状态   剧情状态  伏笔  写作约束   常驻规则
  (L2产出)  (L2维护)  (L2/L3  (L2/L3 (L3产出)  (L1/L2
              │        交互)   交互)             产出)
              │
              ↓
    ┌─────────────────────────────────────────┐
    │              RAG 双重职能                 │
    ├────────────────────┬────────────────────┤
    │ RAG-历史回顾        │ RAG-技法参考        │
    │ 前文章节片段/摘要    │ L3映射关系库       │
    │ 已生成内容索引       │ L4微观技法库       │
    │ 随创作动态增长       │ 预先构建静态库     │
    └────────────────────┴────────────────────┘
```

每次创作时：
- **世界书**提供确定性上下文："主角现在是金丹期，在青云城，与老周是盟友"
- **RAG历史回顾**提供前文锚点："老周上次在第45章出场时，曾暗示京城有秘境线索"
- **RAG技法参考**提供范例素材："类似的金丹期战斗描写有哪些范例"

世界书保证**不犯错**，RAG历史回顾保证**不遗忘**，RAG技法参考保证**写得好**。

## 四、世界书管理员 Agent

### 角色定位

世界书不应是静态文档，而是在创作过程中不断演化的**活体状态机**。需要一个专门的Agent来管理这个演化过程。

**定位**：元层Agent，不直接参与创作，而是在每一轮创作完成后做"后处理"。类似于：

- 数据库的WAL（Write-Ahead Log）——记录变化
- Git的版本控制——维护状态快照和变更历史
- 游戏引擎的状态管理器——所有GameObject的当前状态汇总

### 核心职责

```
创作流水线（L1→L2→L3→L4）
        │
        ↓ 每完成一个序列/章节
        │
┌───────────────────────────────────────┐
│      世界书管理员 Agent                │
├───────────────────────────────────────┤
│ 1. 读取：当前序列/章节的产出           │
│ 2. 提取：新人物、状态变化、新地点、   │
│          新道具、伏笔、关系变化        │
│ 3. 更新：修改对应条目                  │
│ 4. 冲突检测：新内容是否与旧设定矛盾？  │
│ 5. 压缩：旧条目是否需要合并/归档？    │
│ 6. 产出：更新后的世界书 + 变更摘要    │
└───────────────────────────────────────┘
        │
        ↓ 变更摘要反馈
        │
┌───────────────────────────────────────┐
│        L2 专家会议（下一轮）           │
│  架构师查询世界书 → 理解当前世界状态   │
│  编辑查询世界书   → 理解当前情绪进度   │
│  人物设计查询世界书 → 理解人物当前状态 │
└───────────────────────────────────────┘
```

### 与L2专家的关系

世界书管理员**不是第四位专家**。它不参与讨论、不评判创作质量。

- 它更像一个**秘书/档案管理员**
- 专家们创作，管理员记录
- 专家们查询，管理员提供
- 唯一的"主动行为"是冲突检测——发现新产出与旧设定矛盾时**提醒**主编（用户）

### 工作流示例

**场景**：L2完成"拍卖会打脸"序列，产出精修大纲。

```
管理员Agent读取产出：

【提取】
- 新出现人物：鉴定师老周（帮助者）
- 关系变化：主角与老周 无关系→盟友
- 道具变化：晶核已售出，获得300灵石
- 伏笔激活：老周暗示"京城的拍卖会更大"
- 势力状态：主角在青云城声望+50
- 已完成序列：03-拍卖会打脸 → 归档层

【冲突检测】
- ✅ 主角当前金丹期修为 = 之前设定，一致
- ✅ 300灵石收入 = 合理范围
- ⚠️ 警告：上一章主角说"不再相信任何人"，本章却轻易信任老周
  → 反馈给主编确认是否有意为之

【更新】
- 世界书/人物/主角：灵石 50→350，在青云城声望+50
- 世界书/人物/老周：新建条目
- 世界书/伏笔/活跃：新增"京城拍卖会"伏笔，预计回收章节：待定
- 世界书/归档/已完成序列：03-拍卖会打脸 → 压缩为摘要

【产出】
→ 变更摘要发送给下一次L2专家会议的上下文
→ 冲突警告发送给主编（用户）
```

## 五、世界书的自增长结构

### 设计目标

世界书不能无限制膨胀，必须解决三个问题：

1. **规模控制**：200章后世界书可能累积大量条目，不能全量塞入上下文
2. **信息衰减**：第5章的事到第200章只需知道"发生过"而非每个细节
3. **回溯能力**：AI需要理解"角色如何走到今天"，不能只看当前状态

### 四层结构

```
世界书
│
├── 【核心层】永久条目——不随剧情变化
│   ├── 世界观规则
│   │   ├── 力量体系：炼气→筑基→金丹→元婴→化神
│   │   ├── 地理总览：青云城(主角起点)、京城(权力中心)、北境(战场)
│   │   └── 历史大事件：百年前仙魔大战、灵气复苏
│   ├── 核心设定
│   │   ├── 金手指规则：混沌珠每天可产出10灵石
│   │   └── 世界基本法则：凡人不可修仙/灵气浓度地域差
│   └── 阅读契约
│       ├── 类型约束：玄幻爽文，不可太虐
│       └── 毒点红线：禁止圣母、禁止降智
│
├── 【活跃层】当前有效条目——随剧情实时更新
│   ├── 人物状态（每人一条）
│   │   ├── 姓名、行动元（主体/敌对者/帮助者）
│   │   ├── 当前修为、当前地点
│   │   ├── 核心关系：[对主角：忠诚/75]，[对反派A：仇恨/90]
│   │   ├── 心理状态：警惕/自信/绝望
│   │   └── 参与章节：[1-5, 12-18, 30-32]
│   ├── 活跃伏笔
│   │   ├── 伏笔内容
│   │   ├── 埋入章节
│   │   ├── 预计回收章节（可为"待定"）
│   │   └── 状态：待回收/已回收/废弃
│   ├── 当前道具/物品
│   │   ├── 主角持有：[混沌珠, 300灵石, 神秘令牌]
│   │   └── 关键物品下落：[天阶功法残卷: 在反派A手中]
│   └── 势力状态
│       ├── 青云宗：对主角态度 友善/崇拜，实力 中等
│       └── 黑风寨：对主角态度 仇恨，实力 弱（已被灭70%）
│
├── 【归档层】历史快照——保留摘要而非全文
│   ├── 已完成序列摘要
│   │   └── 序列03-拍卖会打脸：主角以金丹期修为震慑全场，
│   │       售出晶核获300灵石，结识鉴定师老周，埋下京城伏笔
│   ├── 已回收伏笔记录
│   │   └── [第5章埋→第30章收] 神秘令牌实为宗门信物，效果：合理
│   ├── 退场人物
│   │   └── 路人甲：筑基期，第10章被反派所杀，作用：展示反派残忍
│   └── 条目版本历史
│       └── 主角修为：[1-30:筑基]→[31-80:金丹]→[81-今:元婴]
│
└── 【索引层】元数据——用于检索和管理
    ├── 条目-章节映射：哪些条目在哪些章节被激活过
    ├── 条目间关系图：人物↔地点↔事件↔道具的关联网络
    └── 触发词索引：哪些词会激活哪些条目
```

### 关键设计原则

#### 1. 版本链而非覆写

角色的修为不是简单地"金丹→元婴"，而是保留完整版本历史：

```
主角修为: [第1-30章: 筑基期（战斗中突破的描述、关键事件摘要）]
          [第31-80章: 金丹期（突破原因：吞噬天材地宝）]
          [第81章至今: 元婴期（突破原因：生死战中顿悟）]
```

这比"主角现在是元婴期"多不了多少token，但对AI理解人物成长弧光至关重要。

#### 2. 重要性衰减

每个条目标记一个权重分数，影响是否注入上下文：

- 初始权重取决于：是否核心角色/核心伏笔/常驻规则
- 每次被创作引用 +1
- 连续10章未被引用 -1
- 权重低于阈值的条目不出现在L4上下文中，但仍保留在活跃层
- 权重归零 → 归档

#### 3. 自动压缩

已完成序列从详细描述压缩为摘要。归档层存储"一句话概括+关键数据"，不存全文：

```
压缩前（活跃层）：
序列03-拍卖会打脸：主角在青云城拍卖行展示晶核，被赵少嘲讽，鉴定师老周
鉴定后发现晶核品级极高，全场震惊，最终以300灵石成交。主角表现淡然，
赵少灰头土脸离开。拍卖后老周私下找到主角，暗示京城有更大机会。

压缩后（归档层）：
序列03-拍卖会打脸[第12-14章]：主角展示实力，获300灵石，结识老周，
埋京城伏笔。打脸对象：赵少（敌对者，后在第45章再次出场）。
```

#### 4. 触发词自维护

管理员Agent监控触发词效果，自动优化：

- 发现某条目的触发词总在不该激活时激活 → 增加触发条件（如需要同时出现两个词）
- 发现某条目应该被激活却未被激活 → 补充触发词
- 发现两个条目的触发词高度重叠，经常同时激活 → 考虑合并

#### 5. 增量注入策略

世界书条目不需要全量注入上下文。注入策略按场景分层：

```
【L2 专家会议】注入：
├── 核心层全部（世界观规则、核心设定、阅读契约）
├── 活跃层中与当前讨论相关的条目（相关人物、相关伏笔、势力状态）
├── 归档层中上一个序列的摘要（理解"刚刚发生了什么"）
└── 变更摘要（自上次会议以来的变化）

【L4 渲染层】注入：
├── 当前章节涉及的人物状态条目
├── 当前场景的地点条目
├── 本章相关的写作约束（视角/节奏/话语模式）
└── 本章需要回收的伏笔（如有）
```

## 六、世界书与RAG的协同工作流

### 三元协同架构

```
                    世界书                    RAG-历史回顾
                  (当前状态)                  (过往记忆)
                      │                          │
                      └──────────┬───────────────┘
                                 │
                    ┌────────────┴────────────┐
                    │     创作层 (L1-L4)       │
                    └────────────┬────────────┘
                                 │
                          RAG-技法参考
                         (写得更好的素材)
```

### 各层的查询策略

| 层 | 世界书 | RAG-历史回顾 | RAG-技法参考 |
|----|--------|-------------|-------------|
| L1 | — | 类似梗概参考 | 类型文趋势参考 |
| L2 | 世界规则、人物状态、当前进度 | 前文未回收伏笔、历史人物关系 | 爽点公式、序列模板 |
| L3 | 写作约束、人物当前状态 | 前文类似场景的叙事处理方式 | 效果→技术映射规则 |
| L4 | 当前场景人物/地点/视角约束 | 前文同一角色/地点的描写方式 | 参考文本范例 |

### 历史回顾RAG的索引策略（与笔记7的衔接）

笔记7的多维度切片（plot/character/emotion/function）天然适合做历史回顾：

```
角色"老周"第300章再出场时：
  ├── character维度 → 检索老周的所有出场片段
  ├── plot维度     → 检索老周参与过的序列摘要
  └── 世界书       → 老周当前状态（修为、关系、地点）
```

检索结果经过Agent压缩后注入上下文，而非原文全量注入。几万字的出场记录压缩为几百字的摘要+关键对白片段。

### 分工总结

| 问题 | 世界书 | RAG-历史回顾 | RAG-技法参考 |
|------|--------|-------------|-------------|
| 主角修为忘了 | ✅ 当前状态 | ✅ 历次突破详情 | — |
| 伏笔忘了回收 | ✅ 活跃伏笔列表 | ✅ 伏笔埋入时的原文 | — |
| 老角色再出场忘了特征 | ✅ 当前状态 | ✅ 历史出场片段 | — |
| 不知道怎么写打脸 | — | ✅ 前文打脸写法保持一致性 | ✅ 爽点公式+范例 |
| 角色OOC | ✅ 核心特性+行为逻辑 | ✅ 历史行为模式 | — |
| 想参考古龙风格 | — | — | ✅ 微观技法库 |
| 世界观矛盾 | ✅ 冲突检测 | ✅ 历史设定细节 | — |

## 七、待讨论问题
