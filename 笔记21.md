- **文档增强有效**：5 视角改写解决了中文专有名词检索问题。"郑屠"从命中错误序列 → 正确命中；"武疯子被枪毙"从无结果 → 正确命中

- **混合检索公式有效**：`vector_score × 0.6 + keyword_score × 0.4`

## 二、RAG 的核心缺陷（笔记6）

### 2.1 四个根本问题

| 问题 | 表现 |
|------|------|
| **叙事完整性破坏** | 固定长度切片把"打脸"序列切成碎片——铺垫在第 2 片、爆发在第 3 片、余韵在第 4 片 |
| **序列逻辑断裂** | 一个完整的因果链（铺垫→冲突→转折→回收）被切散 |
| **边界切割问题** | 关键信息刚好切在边界处（"主角拿出——"断了，下片开头是"——了晶核"） |
| **检索结果失真** | 返回的碎片缺乏上下文，AI 无法理解完整叙事逻辑 |

### 2.2 示例

L2 生成"拍卖会打脸"序列（5000字）：
- 压抑阶段：反派嘲讽、众人轻视（2000字）
- 爆发阶段：主角亮宝、全场震惊（2000字）
- 余韵阶段：天价成交、反派失声（1000字）

传统 1000 字/片 + 200 字重叠 → 5 个碎片。检索"拍卖会打脸"返回的是破碎片段，无法支撑专家决策。

## 三、知识库检索专家（笔记6 核心概念）

### 3.1 定位

不同于简单 RAG（查询→检索→返回），本系统在 RAG 之上增加一层 **Agent 智能加工层**。这个 Agent 称为"知识库检索专家"——它是 L2 专家会议的**参与者**，而非外部工具。

**与普通 RAG 的本质区别**：

| | 普通 RAG | 知识库检索专家 |
|------|---------|-------------|
| 触发方式 | 每次查询被动检索 | 主动理解会议语境，判断是否需要检索 |
| 检索策略 | 固定维度 | 根据当前讨论内容选择维度 |
| 结果处理 | 直接返回原文 | 智能加工（压缩、重组、摘要、推理）后返回 |
| 维度选择 | 全维度 | 剧情架构师查剧情维，人物设计师查人物维 |
| 语境感知 | 无 | 理解会议正在讨论什么，需要什么信息 |

### 3.2 工作流

```
专家会议讨论 → 检索专家监听 → 判断需求
                                ↓
                          选择检索维度
                           (plot/character/emotion/function)
                                ↓
                          查询向量数据库
                                ↓
                          获取标签/指针
                                ↓
                          读取 MD 文本库（完整切片）
                                ↓
                          智能加工处理
                           (压缩/重组/摘要/推理)
                                ↓
                          返回专家会议
```

### 3.3 检索场景示例

**场景**：网络编辑评估"拍卖会打脸的爽点节奏是否合理"

```
检索专家处理：
1. 识别语境：网络编辑发言，关注爽点节奏
2. 选择维度：emotion（情绪维度）
3. 检索结果：
   - 压抑阶段（2000字）+ 标签：压抑强度=中
   - 爆发阶段（2000字）+ 标签：爆发强度=高
   - 余韵阶段（1000字）+ 标签：余韵强度=低
4. 智能加工：
   - 分析情绪曲线：压抑→爆发→余韵
   - 评估节奏：压抑略短，建议延长
5. 压缩为 300 字摘要返回给网络编辑
```

## 四、多维度语义切片

### 4.1 设计思路

放弃机械的重叠切片，采用多维度智能切分。同一份原始文本，从不同角度切分成不同粒度的片段。

| 维度 | 切片单位 | 边界 | 适用专家 |
|------|---------|------|---------|
| **plot**（剧情） | 序列/事件 | 序列开始/结束 | 剧情架构师 |
| **character**（人物） | 角色相关段落 | 人物出场/退场 | 人物设计师 |
| **emotion**（情绪） | 情绪段落 | 情绪转折点 | 网络编辑 |
| **function**（功能） | 叙事功能 | 功能边界 | 剧情架构师 |

### 4.2 原型切片数据（笔记7）

300 章《都市剑说》→ 40 序列 + 91 人物：

| 维度 | 切片数 | 说明 |
|------|--------|------|
| plot | 28 | 按序列切分，每个切片 = 一个完整序列 |
| character | 109 | 按人物拆分，每个切片 = 一个人物的相关段落 |
| emotion | 64 | 按情绪阶段拆分（迷茫/惊讶/从容/霸气...） |
| function | 57 | 按叙事功能拆分（铺垫/转折/获得/冲突...） |
| **总计** | **258** | |

### 4.3 切片示例

序列"魔头苏醒与职场首秀"（5000字）

```
plot 维度:
  └─ 1 个切片: 完整序列

character 维度:
  ├─ 李白切片: 李白的所有戏份
  ├─ 清瑶切片: 清瑶的所有戏份
  ├─ 郑屠切片: 郑屠的所有戏份
  └─ 周真人切片: 周真人的所有戏份

emotion 维度:
  ├─ 迷茫阶段
  ├─ 惊讶阶段
  ├─ 从容阶段
  └─ 霸气阶段

function 维度:
  ├─ 铺垫功能（入场+穿越设定）
  └─ 获得功能（首次展示能力）
```

### 4.4 粒度冲突与解决

不同维度切片粒度不同，存在冲突：
- 人物维度：主角戏份跨多个序列
- 剧情维度：单个序列完整呈现
- 情绪维度：按情绪阶段切分

**解决**：分别返回，侧重不同，检索专家统一处理。查询"主角在拍卖会的表现"时，优先返回人物维度切片，补充剧情维度作为上下文参考。

### 4.5 切片边界判定

关键设计：**切片器不依赖 AI 推理**。L2 的多维标注已经提供了序列边界和功能标签，切片器只是按标注切分文本。过程是确定性的。

## 五、文档增强

### 5.1 五视角改写

每个序列存入向量库时，生成 5 种视角的改写文本。目的：**增加检索命中入口**。

| 视角 | 内容 | 检索命中场景 |
|------|------|------------|
| **情节概述** | 发生了什么，怎么解决的 | 专家想查"之前类似情节" |
| **爽点分析** | 为什么读者觉得爽，用了什么手法 | 网络编辑评估节奏 |
| **人物关系** | 人物之间的互动和关系变化 | 人物设计师查角色历史 |
| **情感弧线** | 读者情绪变化过程 | 编辑查情绪曲线 |
| **读者问题** | 读者可能会问的 3-5 个问题 | 查伏笔回收状态 |

### 5.2 原型验证效果（笔记7）

| 查询 | 无增强 | 有增强 |
|------|--------|--------|
| "郑屠" | seq_015（错误）| seq_001（正确）|
| "武疯子被枪毙" | 无结果 | seq_021（正确）|
| "戏弄道士" | 无结果 | seq_006（正确）|

**原理**：同一信息的 5 种表达 = 5 倍的检索命中入口。"郑屠"这个名字在正文中只出现一次，但经过 5 视角改写后，在"情节概述""人物关系""读者问题"中都被提及，embedding 向量更容易匹配。

### 5.3 增强时机

**不是在查询时增强，而是在索引时增强**。每个序列入库时执行一次 5 视角改写，查询时直接检索已有的增强文本。这样：
- 查询延迟不受影响（增强已完成）
- 增强质量有保证（LLM 调用，非实时生成）
- 同一个序列的增强结果可复用

### 5.4 检索文本构建公式（笔记7 实战）

```
search_text = 多视角改写文本 + 完整内容的前 500 字
```

存入向量库的 search_text 包含两部分：增强摘要（多视角改写）用于语义匹配，内容前 500 字用于关键词匹配。完整切片存在 MD 文本库中，检索命中后按需读取。

## 六、混合检索

### 6.1 为什么需要混合

向量检索对中文专有名词效果不佳（人名、功法名、地名）。"郑屠"的 embedding 向量可能与"屠夫""屠城"更接近，而非真正相关的"魔头苏醒序列"。

**解决方案**：向量搜索 + 关键词过滤。

### 6.2 混合得分公式（笔记7 实战验证）

```python
vector_score = 1 - min(distance / 1.5, 1)   # 余弦距离转 0-1 得分
keyword_score = matched_keywords / total_keywords  # 关键词匹配率
hybrid_score = vector_score × 0.6 + keyword_score × 0.4
```

### 6.3 原型检索效果

| 查询 | 命中序列 | 得分 | 说明 |
|------|---------|------|------|
| 郑屠 | 魔头苏醒与职场首秀 | 0.614 | 专有名词命中 |
| 武疯子被枪毙 | 高速惊魂与田野对决 | 0.253 | 模糊描述命中 |
| 戏弄道士 | 直播斗法与当众处刑 | 0.291 | 动作描述命中 |
| 穿越归来 | 魔头苏醒与职场首秀 | 0.294 | 概念命中 |
| 打脸爽点 | 职场争锋与熊孩子克星 | 0.333 | 类型标签命中 |

### 6.4 检索流程

```
用户查询 "老周在第3章拍卖会上的表现"
    │
    ├── 向量检索（ChromaDB）:
    │   查询 embedding → cosine 相似度 → top 20
    │
    ├── 关键词检索:
    │   提取查询中的专有名词（"老周""拍卖会"）
    │   → keyword_score 匹配
    │
    ├── 混合排序:
    │   合并去重 → hybrid_score → top K (默认 5)
    │
    └── 双库分离:
        向量库结果 → 拿到 slice ID
            │
            ▼
        MD 文本库 → 读取完整切片内容
            │
            ▼
        返回（含完整上下文）
```

## 七、双库分离

### 7.1 设计

| 库 | 存储内容 | 大小 | 速度 | 作用 |
|----|---------|------|------|------|
| **ChromaDB 向量库** | search_text（增强摘要 + 前 500 字）+ embedding 向量 | ~3.5MB / 300章 | 毫秒级 | 快速定位相关切片 |
| **MD 文本库** | 完整切片文本 | ~500KB / 300章 | 文件读取 ~1ms | 提供完整上下文 |

### 7.2 原型存储统计（笔记7）

- 300 章小说 → 258 个切片 → ChromaDB 3.5MB + MD 文本 ~500KB
- 每个切片：embedding 1024 维 × float32 = 4KB + 元数据
- 检索延迟：向量搜索 ~50ms + 文件读取 ~1ms = ~51ms

## 九、自动增量索引

### 9.1 触发时机

Pipeline 中每完成一个序列/章节后，自动触发索引：

| 触发事件 | 索引内容 | 维度 |
|---------|---------|------|
| L2 章纲完成 | 章纲的功能序列描述 | plot, function |
| L3 草稿完成 | 带标签草稿 | plot, character, emotion, function |
| L4 正文完成 | 精修正文 | plot, character, emotion, function |

### 9.2 索引流水线

```
序列/章节完成
    │
    ▼
切片器（确定性，基于 L2 标注边界）
    │
    ├── plot 切片 → collection_plot
    ├── character 切片 → collection_character
    ├── emotion 切片 → collection_emotion
    └── function 切片 → collection_function
    │
    ▼
文档增强（LLM 5 视角改写）→ 构建 search_text
    │
    ▼
关键词索引（同步，秒级）→ 即时可用
    │
    ▼
向量索引（异步，ChromaDB embedding）→ ~1 分钟
```

### 9.3 索引状态标记

```
'keyword_only' → 关键词索引完成，可被关键词检索命中
'embedding'    → 向量索引完成，精确检索可用
'enhanced'     → 5 视角增强完成
'complete'     → 全部完成
```

## 十、Agent+RAG 分层架构（笔记6 §七）

笔记6 提出了统一的 Agent+RAG 架构，将知识库检索专家应用到 L2/L3/L4 三个创作层：

```
┌─────────────────────────────────────────────────────┐
│                   创作模型                            │
├─────────────────────────────────────────────────────┤
│  L1 种子层  →  L2 架构层  →  L3 叙事层  →  L4 渲染层  │
│                  ↓              ↓             ↓     │
│          检索专家(剧情)   检索专家(映射)   检索专家(技法) │
│                  ↓              ↓             ↓     │
│           ┌──────┴──────────────┴─────────────┴───┐ │
│           ↓           Agent 智能加工层             ↓ │
│           └──────┬──────────────┬─────────────┬───┘ │
│                  ↓              ↓             ↓     │
│           向量库(剧情)    向量库(映射)    向量库(技法)  │
│                  ↓              ↓             ↓     │
│           MD 文本库       MD 文本库       MD 文本库    │
└─────────────────────────────────────────────────────┘
```

### 10.1 L2 剧情检索专家

负责从历史章节中检索相关序列和情节。不同专家检索不同维度：
- 剧情架构师 → plot, function
- 人物设计师 → character
- 网络编辑 → emotion

### 10.2 L3 映射检索专家

L3 的核心任务是效果→技术的确定性映射。映射检索专家的职责：
- 理解当前细纲的具体语境
- 判断需要哪些映射规则（视角？节奏？话语模式？）
- 检索 L3 映射关系库
- 多条规则冲突时，根据场景选择/组合
- 返回组合后的映射指令

**与普通 RAG 的区别**：多条映射规则可能冲突，Agent 需要理解场景后做仲裁，而不是简单返回所有匹配结果。

### 10.3 L4 技法检索专家

L4 需要具体的写作范例作为参考。技法检索专家的职责：
- 读取 L3 产出的细纲指令（如"外聚焦 + 短句 + 动作描写"）
- 检索 L4 微观技法库
- 筛选风格一致的参考文本
- 避免参考文本"带偏"生成方向

### 10.4 待讨论的问题

- L3 映射检索专家：多条映射规则冲突时如何仲裁？映射规则是否需要"适用条件"标签？
- L4 技法检索专家：如何判断"风格一致性"？如何避免参考文本"过度影响"生成？
- 三个检索专家是否需要共享信息（如当前故事的整体风格基调）？
- Agent 智能加工层的具体逻辑：压缩？重组？摘要？推理？

## 十一、检索器实现层次

```
HybridRetriever          ← 混合检索（向量 + 关键词 + 双库）
    │
    ├── ChromaRetriever   ← 向量检索（ChromaDB + embedding API）
    │
    └── KeywordRetriever  ← 关键词检索（BM25 / TF-IDF）
```

**SimpleVectorRetriever**（现有 `services/rag.ts`）降级为 `KeywordRetriever` 的默认实现。

**embedding 后端可切换**（笔记7 已验证）：
- DashScope API：`text-embedding-v3`，1024 维，在线
- 本地模型：`qwen3-embedding-4b`（GGUF 量化），离线

## 十二、Pipeline 注入集成

### 12.1 注入时机

专家 Worker 构建 `ExpertContext` 时，根据节点配置决定是否注入 RAG 检索结果。

查询模式：

| 模式 | 行为 | 适用场景 |
|------|------|---------|
| **auto** | 每次发言前自动检索 | 历史回顾——每轮都需要参考前文 |
| **manual** | 用户手动输入查询 | 特定问题查询 |
| **trigger** | 检测到特定关键词触发 | 角色名/地名出现时查询 |

### 12.2 上下文预算

每个 RAG 实例有 `maxInjectionTokens`（默认 2000）：

```
检索到 5 条结果 → 按得分排序 → 逐条累加 token 数
    → 超过 budget 则截断
    → 拼接为:
      ## RAG: {实例名}
      {slice_content_1}
      ---
      {slice_content_2}
```

此预算与世界书注入预算分开管理。

### 12.3 专家维度偏好

| 专家 | 默认检索维度 | 理由 |
|------|------------|------|
| 剧情架构师 | plot, function | 关注剧情结构和因果链 |
| 人物设计师 | character | 关注角色行为和关系 |
| 网络编辑 | emotion | 关注爽点节奏和读者反应 |
| 资深作者 | plot, character | 关注整体一致性 |
| 读者代表 | emotion | 关注体验流畅度 |

## 十五、业界方案对比（笔记6 §八）

| 方案 | 切片策略 | 索引结构 | 检索处理 | 适用场景 |
|------|---------|---------|---------|---------|
| Multi-Vector Retriever | 机械切分 | 摘要+原文分离 | 直接返回 | 通用 |
| Parent Document | 小切大返回 | 层级索引 | 直接返回 | 通用 |
| GraphRAG（微软） | 知识图谱 | 图结构+社区摘要 | 图遍历 | 复杂推理 |
| **本文方案** | 多维度语义切分 | 双库 + Agent 智能加工 | Agent 理解语境后加工 | 网文创作 |

### 本文方案与业界方案的异同

- **Multi-Vector Retriever**：都使用"检索内容≠返回内容"的双库模式。但本文增加了多维度切分和 Agent 智能加工
- **Parent Document Retriever**：都解决切片边界问题。但本文按叙事边界而非机械长度切分
- **GraphRAG**：都关注实体关系和层级。但 GraphRAG 偏通用知识抽取，本文针对网文创作有"爽点""功能""序列"等特定维度

### 本文方案的核心创新

1. **叙事学驱动的切分维度**：按 plot/character/emotion/function 切分，而非机械长度
2. **Agent 作为检索专家**：主动理解语境，而不是被动响应查询
3. **与创作流程深度绑定**：L2/L3/L4 各有专门的检索专家，检索维度与专家职责对齐

## 十七、RAG 配置页面设计

### 17.1 多 RAG 机制选择

系统提供三种检索机制，用户可按实例选择：

| 机制 | 标识 | 原理 | 适用场景 | 依赖 |
|------|------|------|---------|------|
| **简单关键词** | `keyword` | 基于 TF-IDF 的关键词匹配，无 embedding | 小数据量（< 50 章）、快速原型 | 无外部依赖 |
| **混合检索** | `hybrid` | 向量相似度 × 0.6 + 关键词 × 0.4 | 生产环境首选，平衡语义与精确 | ChromaDB + embedding API |
| **纯向量** | `vector` | 纯 ChromaDB embedding 相似度 | 语义相关性优先的场景 | ChromaDB + embedding API |

创建实例时选择机制，后续可切换（切换时重建索引）。

### 17.2 设计思路：机制驱动页面

世界书的所有书共享同一套 Schema（条目 = keys + content + 选择性逻辑），所以一个页面可以管理所有书。但 RAG 的三种机制本质不同：

| | keyword | hybrid | vector |
|------|---------|--------|--------|
| 配置项 | ~3 个 | ~12 个 | ~8 个 |
| 索引展示 | 文档列表 | 维度表格 + 增强状态 | 维度表格 |
| 检索测试 | 匹配词高亮 | 向量分+关键词分拆解 | 向量相似度 |
| 导入格式 | 纯文本/JSON | JSON + 预生成切片目录 | JSON + 已有 embedding |
| 外部依赖 | 无 | ChromaDB + embedding API | ChromaDB + embedding API |

因此设计为**实例选择器常驻顶部，下方根据选中实例的机制类型，渲染对应的配置页面组件**。

```
RAGManager.vue（路由入口）
    │
    ├── 实例栏（常驻）— 下拉切换实例 + 新建/删除/导入/导出
    │
    └── 根据 instance.retrieverType 动态渲染：
        ├── 'keyword'  → KeywordRAGConfig.vue
        ├── 'hybrid'   → HybridRAGConfig.vue
        └── 'vector'   → VectorRAGConfig.vue
```

每个机制组件内部使用自己的 Tab 结构。共享的部分（检索日志）提取为独立组件 `RAGSearchLog.vue`。

    │   └── 选择切片目录（仅 hybrid/vector）
    │
    ▼
根据实例机制分流处理：
    │
    ├── keyword:
    │   JSON 文档 → 解析 → 写入 Map → 存 JSON
    │
    ├── hybrid:
    │   JSON/目录 → 检测增强文本是否存在
    │   ├── 有增强 → 直接切片入库
    │   └── 无增强 → 标记 "待增强" → 后台 LLM 5视角改写
    │   → 检测 embedding 是否存在
    │   ├── 有 → 直接写入 ChromaDB
    │   └── 无 → 标记 "待向量化" → 后台 embedding API 批量处理
    │
    └── vector:
        JSON/目录 → 检测 embedding 是否存在
        ├── 有 → 直接写入 ChromaDB
        └── 无 → 后台 embedding API 批量处理
```
