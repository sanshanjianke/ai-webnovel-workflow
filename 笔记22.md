# 笔记22：预处理节点设计

## 一、背景

当前工具栏有 📥输入源、📦群聊、⚖️判断、📤输出、👁📖世界书管理员、🔍RAG 六种节点。用户如果想"拆书"——把一整本小说拖入系统分析——输入源会直接把全文作为一个 PipelineObject 塞给下游专家，必爆 AI 上下文窗口。

需要一个**预处理节点**，放在输入源和专家之间，把大文件拆成可处理的单元再流入下游。

## 二、核心设计决策

### 2.1 预处理是一个独立的模块文件，不是简单的拆分节点

拆分只是预处理管线中的一个阶段。完整管线：

```
原始文本 → [替换/清洗] → [拆分] → [窗口合并] → 输出 N 个分块
```

每个阶段是可插拔的函数，用户可编排顺序、可自己写代码。

### 2.2 AI 语义拆分作为独立的可选 Agent

不是预处理节点的一部分，而是画布上一个独立的专家节点。用户连线：`预处理(粗拆) → AI语义拆分Agent(精修边界) → 下游专家`。两个节点解耦，各自可独立使用。

### 2.3 拆分策略以正则为主，token/行数为辅

网文格式多变（第x章、第x节、分页、无规律），正则表达式最灵活。提供预设模板覆盖常见格式，用户可自定义。

### 2.4 窗口合并：通用模型 `{n-i, ..., n, ..., n+i}`

拆分后的分块可按需合并。n 是主处理单元（可包含多个拆分块），i 是每侧上下文块数。

例如：一本书拆成 50 章，设 `n=5`（5章一组处理），`i=2`（前后各带2章上下文），则每组共 9 章，其中第 3-7 章为主处理对象，第 1-2 章和第 8-9 章为上下文辅助。两个参数均可独立配置。

## 四、预处理核心模块 (`preprocessor.ts`)

### 4.1 核心数据结构

```typescript
// 文本段——管线中流动的基本单元
interface TextSegment {
  content: string;
  meta: {
    index: number;          // 当前序号
    title?: string;         // 章节标题（拆分时自动提取）
    chapterNumber?: number; // 章节号
    charCount: number;      // 字符数
    // 用户自定义扩展字段
    [key: string]: any;
  };
}

// 阶段类型枚举
type StageType = 'replace' | 'split_regex' | 'split_token' 
               | 'split_lines' | 'window_group' | 'custom';

// 单个阶段配置
interface StageConfig {
  id: string;              // 唯一标识
  type: StageType;
  enabled: boolean;
  label: string;           // 用户可读名称
  config: Record<string, any>;  // 阶段特定参数
  customCode?: string;     // type='custom' 时的用户代码
}

// 完整预处理配置（存在节点 data 中）
interface PreprocessConfig {
  stages: StageConfig[];
  outputMode: 'expand' | 'pack';
  // 'expand': 1→N 分流，每个分块独立成为 PipelineObject 流向下游
  // 'pack':   N→1 归并，所有分块打包进一个 PipelineObject（多文件），下游收到一个对象
}
```

### 4.2 阶段接口与内置实现

每种阶段实现同一个接口：

```typescript
interface PreprocessStage {
  name: string;
  execute(segments: TextSegment[], config: Record<string, any>): TextSegment[];
}
```

**内置阶段**：

| 阶段 | 类型 | 功能 | 关键配置 |
|------|------|------|---------|
| `RegexReplaceStage` | `replace` | 正则替换/清洗文本 | `pattern`, `replacement`, `flags` |
| `RegexSplitStage` | `split_regex` | 按正则匹配边界拆分 | `pattern`, `preset`(预设名), `includeSeparator` |
| `TokenSplitStage` | `split_token` | 按 token 数拆分 | `maxTokens`, `overlap`, `encoding`(默认中文字符估算) |
| `LineSplitStage` | `split_lines` | 按行数拆分 | `maxLines`, `overlap` |
| `WindowGroupStage` | `window_group` | 合并相邻分块为滑动窗口 | `groupSize`(主单元块数n，默认5), `contextBefore`(前文块数i，默认2), `contextAfter`(后文块数i，默认2) |
| `CustomStage` | `custom` | 用户自定义代码 | `customCode`(JS 函数体字符串) |

### 4.4 管线执行器

```typescript
function runPreprocessPipeline(
  inputText: string,
  stages: StageConfig[]
): TextSegment[] {
  // 1. 初始段：整个文本作为一个 segment
  let segments: TextSegment[] = [{
    content: inputText,
    meta: { index: 0, charCount: inputText.length }
  }];

  // 2. 按顺序执行每个启用的阶段
  for (const stage of stages) {
    if (!stage.enabled) continue;
    const executor = getStageExecutor(stage.type);
    segments = executor.execute(segments, stage.config);
  }

  // 3. 重新编号
  segments.forEach((s, i) => { s.meta.index = i; });
  
  return segments;
}
```

#### 环境检测

// preprocessor.ts 启动时调用
async function detectAvailableRuntimes(): Promise<Record<string, boolean>> {
  const result = { python: false, perl: false, javascript: true }; // JS 始终可用

#### 数据传递协议

所有语言共享同一套输入输出协议——**JSON via stdin/stdout**。关键设计：不通过命令行参数传递数据（避免转义问题和长度限制），脚本写入临时文件后执行，stdin/stdout 用管道传递 JSON。

```
输入 (stdin):  { "segments": [{"content":"...", "meta":{...}}, ...], "config": {...} }
输出 (stdout): { "segments": [{"content":"...", "meta":{...}}, ...] }
错误 (stderr): 逐行实时推送到前端
```

#### 执行基础设施

三个问题的解决：

**① 命令行传脚本 → 改为临时文件**：`spawn('python3', ['-c', 脚本])` 在脚本包含引号、换行、缩进混合时极易出错，且命令行长度有 OS 上限（Linux 2MB，但实际远小于此）。改为将包装后的完整脚本写入临时文件，`spawn('python3', [临时文件路径])`。执行完毕后清理临时文件。

**② stdin 死锁预防**：OS 管道缓冲区默认 64KB。大数据下父进程写 stdin 和子进程读 stdin 可能互相等待。解决方案：父进程先写完所有 stdin，调用 `stdin.end()` 关闭写端，然后再开始读 stdout。Node.js 的 `child_process.spawn` 返回的 `child.stdin` 是 WritableStream，调用 `.end()` 即可。

```typescript
// 安全的子进程调用模式
const child = spawn(cmd, [tmpFile], {
  stdio: ['pipe', 'pipe', 'pipe'],
  env: { ...process.env, PYTHONIOENCODING: 'utf-8' }
});

// 写入 stdin 并立即关闭写端
child.stdin.write(inputJson);
child.stdin.end();  // ← 关键：关闭写端，防止子进程阻塞等待

// 然后读 stdout
const chunks: Buffer[] = [];
child.stdout.on('data', (chunk) => chunks.push(chunk));

// stderr 实时收集
let stderr = '';
child.stderr.on('data', (data) => { stderr += data.toString(); });
```

**③ 编码**：Python 需要设置 `PYTHONIOENCODING=utf-8` 环境变量（或模板中用 `sys.stdin.buffer.read().decode('utf-8')`）。Perl 模板中显式 `binmode(STDIN, ':utf8')` + `binmode(STDOUT, ':utf8')`。JS 原生支持 UTF-8，无额外处理。

#### 辅助函数（所有语言统一提供）

三种语言都提供等价的辅助函数，消除接口不对称：

```
            JS                        Python                   Perl
──────────────────────────────────────────────────────────────────────────
创建段      utils.segment(c, m)       segment(content, meta)   segment($content, \%meta)
正则匹配    utils.regex(p, t)         regex_match(p, t)        regex_match($p, $t)
Token估算   utils.countTokens(t)      count_tokens(t)          count_tokens($t)
```

辅助函数在系统包装模板中自动生成，用户可直接调用。

#### Python（CLI 子进程）

```python
# 系统生成的完整包装脚本（用户只写 "用户代码" 标记之间的部分）

import json, sys, re

# ═══ 系统生成的辅助函数 ═══
def segment(content, meta=None):
    m = meta.copy() if meta else {}
    m['charCount'] = len(content)
    return {'content': content, 'meta': m}

def regex_match(pattern, text):
    return [m.group() for m in re.finditer(pattern, text)]

def count_tokens(text):
    return len(text)  # 中文字符估算
# ═══ 辅助函数结束 ═══

def main():
    # 显式 UTF-8 解码，避免系统编码问题
    raw = sys.stdin.buffer.read()
    data = json.loads(raw.decode('utf-8'))
    segments = data['segments']
    config = data['config']

    # ═══════ 用户代码开始 ═══════

    result = []
    for s in segments:
        parts = re.split(r'---+场景分隔线---+', s['content'])
        for i, part in enumerate(parts):
            if part.strip():
                result.append(segment(part, {**s['meta'], 'sceneIndex': i}))

    # ═══════ 用户代码结束 ═══════

    output = json.dumps({'segments': result}, ensure_ascii=False)
    sys.stdout.buffer.write(output.encode('utf-8'))

if __name__ == '__main__':
    main()
```

#### Perl（CLI 子进程）

```perl
# 系统生成的完整包装脚本

use strict;
use warnings;
use utf8;
binmode(STDIN, ':utf8');
binmode(STDOUT, ':utf8');

# ═══ 系统生成的辅助函数 ═══
# 检测 JSON 模块
my $json_ok = eval { require JSON; JSON->import(qw(decode_json encode_json)); 1 };

sub segment {
    my ($content, $meta) = @_;
    $meta //= {};
    $meta->{charCount} = length($content);
    return { content => $content, meta => $meta };
}

sub count_tokens {
    return length(shift);
}
# ═══ 辅助函数结束 ═══

die "Perl JSON module not installed. Run: cpan JSON\n" unless $json_ok;

my $json_text = do { local $/; <STDIN> };
my $data = decode_json($json_text);
my $segments = $data->{segments};
my $config = $data->{config};

# ═══════ 用户代码开始 ═══════

my @result;
for my $s (@$segments) {
    my @parts = split(/---+场景分隔线---+/, $s->{content});
    for my $i (0..$#parts) {
        next unless $parts[$i] =~ /\S/;
        my %meta = %{$s->{meta}};
        $meta{sceneIndex} = $i;
        push @result, segment($parts[$i], \%meta);
    }
}

# ═══════ 用户代码结束 ═══════

print encode_json({ segments => \@result });
```

#### CLI 调用安全措施

- **执行方式**：脚本写入临时文件，`spawn(cmd, [tmpFile])` 执行，避免命令行注入和长度限制。执行后清理临时文件
- **编码保证**：Python 用 `sys.stdin.buffer` + `sys.stdout.buffer` 直接读写字节流并显式 UTF-8 编解码；Perl 用 `binmode` 设置 UTF-8；环境变量追加 `PYTHONIOENCODING=utf-8` 作为双保险
- **stdin 死锁预防**：父进程写完 stdin 立即 `.end()` 关闭写端，再收集 stdout。stdout 通过 `sizeLimitStream` 包装，超 50MB 自动截断
- **超时限制**：30 秒超时，超时自动 SIGKILL
- **依赖检测**：Perl 模板中 `eval { require JSON }` 运行时检测，未安装时向前端返回明确错误信息（"Perl JSON 模块未安装，请执行 cpan JSON"）。Python 标准库自带 json，无需额外检测
- **错误报告**：stderr 实时推送到前端（SSE 事件），非零退出码 + stderr + stdout 片段一并返回，不静默吞错

## 五、后端集成

### 5.1 新增 Worker (`pipeline-workers.ts`)

```typescript
export async function preprocessorWorker(
  nodeId: string,
  inputQueue: AsyncQueue<PipelineObject>,
  downstreamQueues: Map<string, AsyncQueue<PipelineObject>>,
  emit: (event: PipelineEventV2) => void,
  state: WorkerSharedState,
  edgePortMap: Map<string, string> = new Map()
): Promise<void> {
  const config = state.nodeConfigs.get(nodeId);
  const preprocessConfig: PreprocessConfig = config.preprocessConfig;
  
  while (!state.stopRequested) {
    await waitIfPaused(state);
    const obj = inputQueue.dequeueSync();
    if (!obj) { await sleep(1000); continue; }

    // 获取输入文本（合并所有 input 文件）
    const inputText = obj.files
      .filter(f => f.category === 'input')
      .map(f => f.content).join('\n\n');

    // 运行预处理管线
    const segments = runPreprocessPipeline(inputText, preprocessConfig.stages);

    // 根据输出模式分发
    if (preprocessConfig.outputMode === 'pack') {
      // ── 归并模式 (N→1)：所有分块打包进一个对象 ──
      const packedObj = createPipelineObject(
        `${obj.name}_预处理结果`,
        segments.map(seg => ({
          path: `${seg.meta.title || `chunk_${seg.meta.index}`}.md`,
          content: seg.content
        }))
      );
      packedObj.meta = { ...obj.meta, totalChunks: segments.length, outputMode: 'pack' };
      emit({ type: 'object_init', data: { objectId: packedObj.id, objectName: packedObj.name } });
      for (const [, downQueue] of downstreamQueues) {
        await downQueue.enqueue(packedObj);
      }
    } else {
      // ── 分流模式 (1→N)：每个分块独立为 PipelineObject ──
      for (const seg of segments) {
        const chunkObj = createPipelineObject(
          seg.meta.title || `${obj.name}_chunk_${seg.meta.index}`,
          [{ path: `${obj.name}_chunk_${seg.meta.index}.md`, content: seg.content }]
        );
        chunkObj.meta = { ...obj.meta, chunkIndex: seg.meta.index, totalChunks: segments.length, outputMode: 'expand' };
        emit({ type: 'object_init', data: { objectId: chunkObj.id, objectName: chunkObj.name } });
        for (const [, downQueue] of downstreamQueues) {
          await downQueue.enqueue(chunkObj);
        }
      }
    }
  }
}
```

两种输出模式：

| 模式 | `outputMode` | 1 个上游对象 → | 下游收到 | 适用场景 |
|------|-------------|---------------|---------|---------|
| **分流** | `'expand'` | N 个 PipelineObject | 每个分块独立流转 | 逐章 AI 分析、逐段翻译 |
| **归并** | `'pack'` | 1 个 PipelineObject（含 N 个文件） | 一个对象包含全部分块 | 纯拆分存档、拆完交给其他工具 |

归并模式的优势：如果用户工作流只做拆分（不用下游专家逐块处理），所有分块打包在一个对象里，后续可以直接下载、导入文档库、或经由输出节点收集。不会产生 N 个重复的 PipelineObject 挤满下游队列。

### 6.4 预设模板

用户可以一键加载预设的阶段组合：

| 模板 | 阶段序列 | 适用 |
|------|---------|------|
| 网文拆书(标准) | 替换(去广告) → 按正则拆分(第X章) → 窗口合并(n=5,i=2) | 中文网文逐章分析 |
| 网文拆书(简单) | 按正则拆分(第X章) | 仅拆分，无额外处理 |
| 长文通用拆分 | 按 token 拆分(3000/重叠200) → 窗口合并(n=1,i=1) | 无章节标记的长文 |
| 按行拆分 | 按行拆分(每100行) | 剧本/诗歌 |
| 空白模板 | 无预设阶段 | 用户自行搭建 |

## 七、数据流

```
                    预处理节点内部管线
                    ═══════════════
输入源              ┌─ 阶段1: 替换 ───┐
  │                 │  (去广告/规范化)  │
  │  1个对象        └────────┬─────────┘
  │  (全书.txt)              │
  ├────────────────→  ┌──────▼─────────┐
                     │ 阶段2: 拆分      │
                     │  (按第X章拆分)   │  → N个分块
                     └──────┬─────────┘
                            │
                     ┌──────▼─────────┐
                     │ 阶段3: 窗口合并  │
                     │ n=5,i=2,共9块  │  → N个带上下文的组
                     └──────┬─────────┘
                            │
                  ┌─────────┴──────────┐
                  │ 输出模式           │
                  │                    │
                  │ 分流(expand):      │
                  │   N个PipelineObject │  → 下游逐块处理
                  │                    │
                  │ 归并(pack):        │
                  │   1个PipelineObject │  → 下游收到包含全部
                  │   (含N个文件)       │     分块的单个对象
                  └────────────────────┘
```

## 八、AI 语义拆分的分离设计

AI 语义拆分不放入预处理节点，而是一个独立的专家 Agent。用户可选连线：

```
输入源 → 预处理(粗拆) → AI语义拆分Agent → 下游专家
```

AI 语义拆分 Agent 的职责：
- 接收粗拆分后的分块
- 用 LLM 判断相邻分块的语义边界是否合理
- 输出调整后的分块

好处：
- 预处理节点纯确定性，不依赖 LLM，无费用
- AI 语义 Agent 可作为独立专家复用
- 用户可以只用预处理不用 AI，也可以只用 AI 不用预处理
- 两个模块各自独立迭代
