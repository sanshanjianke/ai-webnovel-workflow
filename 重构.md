## 一、当前问题

### 1.1 管道模式不是真正的流水线

**问题描述：**
- 当前实现是"批处理"模式，不是"流水线"模式
- 作者处理完文件1后，要等所有文件都处理完，才开始处理读者的文件
- 用户期望：文件1给作者 → 作者完成 → 文件1给读者，同时作者开始处理文件2

**根本原因：**
- Python 的同步生成器（Generator）无法实现真正的生产者-消费者模式
- `run_pipeline` 方法和 `pipeline_stream` 函数都在遍历文件，导致双重遍历

**期望的流水线模式：**
```
文件1: 作者处理 → 读者处理 → 完成
文件2: 作者处理 → 读者处理 → 完成
文件3: 作者处理 → 读者处理 → 完成

而不是：
文件1: 作者处理
文件2: 作者处理
文件3: 作者处理
文件1: 读者处理
文件2: 读者处理
文件3: 读者处理
```

### 1.2 队列显示延迟

**问题描述：**
- 左侧聊天窗口的队列，要等到第一个文件完全处理完才显示
- 用户期望：第一个文件开始处理时就应该显示队列

**可能原因：**
- SSE 事件传输过程中有缓冲
- 前端事件处理逻辑有问题

### 1.3 按钮状态丢失

**问题描述：**
- 返回画布页面后，暂停/中止按钮变回"开始流程"
- 运行状态没有正确持久化

**已尝试的解决方案：**
- 使用 sessionStorage 持久化状态
- 通过 BroadcastChannel 同步状态

### 2.3 核心设计：生产者-消费者模式

**Node.js 的优势：**
- 原生支持 async/await 和异步迭代器
- 可以轻松实现真正的生产者-消费者模式
- SSE 流式输出简单直接

**流水线实现思路：**

```typescript
// 每个节点有自己的输入队列
const nodeQueues: Map<string, Queue<FileTask>> = new Map()

// 生产者：从上游节点获取输出，放入下游节点的队列
async function producer(upstreamNode: string, downstreamNode: string) {
  for await (const output of upstreamNode.outputs) {
    await nodeQueues.get(downstreamNode).enqueue(output)
  }
}

// 消费者：从自己的队列中取出任务处理
async function consumer(nodeId: string) {
  for await (const task of nodeQueues.get(nodeId)) {
    const output = await processNode(nodeId, task)
    // 将输出传递给下游
    for (const downstream of getDownstreamNodes(nodeId)) {
      await nodeQueues.get(downstream).enqueue(output)
    }
  }
}
```

**真正的流水线执行：**
```
时间轴:
t0: 作者开始处理文件1
t1: 作者完成文件1，读者开始处理文件1，作者开始处理文件2
t2: 读者完成文件1，作者完成文件2，读者开始处理文件2，作者开始处理文件3
t3: 读者完成文件2，作者完成文件3
t4: 读者完成文件3
```

```javascript
const response = await fetch(url, { method: 'POST', body: JSON.stringify(data) })
const reader = response.body.getReader()
const decoder = new TextDecoder()

while (true) {
  const { done, value } = await reader.read()
  if (done) break
  buffer += decoder.decode(value, { stream: true })
```

| `queue_item_complete` | 文件处理完成 | `{index, total}` |
| `queue_complete` | 队列完成 | `{total}` |
| `level_start` | 层级开始 | `{level, total_levels, nodes, file_index}` |
| `level_complete` | 层级完成 | `{level, file_index}` |
| `pipeline_complete` | 管道完成 | `{output}` |
| `pipeline_stopped` | 管道停止 | `{node_id}` |

### 10.3 输出数据结构

```typescript
interface PipelineOutput {
  node_outputs: Record<string, string>  // node_id -> 内容
  total_speeches: number
  meeting_summary: string  // 所有节点输出的汇总
}
```

## 十一、前端需要配合的修改

### 11.1 API 地址

如果 Node.js 后端使用不同端口（如 3000），需要修改 `frontend/vite.config.js`:

```javascript
export default defineConfig({
  server: {
    proxy: {
      '/api': 'http://localhost:3000'  // 改为 Node.js 端口
    }
  }
})
```

### 11.2 事件格式

确保 Node.js 后端发送的 SSE 事件格式与前端期望一致：

```javascript
// 格式一：event: + data:
res.write(`event: ${eventType}\ndata: ${JSON.stringify(data)}\n\n`)

// 格式二：data: + type in JSON
res.write(`data: ${JSON.stringify({ type: eventType, ...data })}\n\n`)
```

### 11.3 BroadcastChannel 事件

前端通过 BroadcastChannel 同步状态，需要确保后端发送的事件类型与前端处理的一致：

```javascript
// 前端期望的事件类型
'queue_state'      // 队列状态
'queue_start'      // 队列开始
'queue_item_start' // 文件开始
'expert_start'     // 专家开始
'chunk'            // 流式内容
'message'          // 完整消息
'done'             // 完成
'running_state'    // 运行状态
```
