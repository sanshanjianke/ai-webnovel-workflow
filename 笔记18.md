## 一、核心架构：分布式工作节点模型

### 1.0 每个节点是独立 Worker

编排层的根本设计理念：**每个节点（专家/群聊）是一个独立的工作节点（Worker），维护自己的任务队列，独立并行运行。**

```
     输入源                  资深作者                   读者代表                输出节点
  ┌──────────┐         ┌──────────────┐         ┌──────────────┐         ┌──────────┐
  │ 任务队列  │  ───→   │   任务队列    │  ───→   │   任务队列    │  ───→   │ 任务队列  │
  │ [对象1,  │         │ [对象1,      │         │ [对象1,      │         │ [对象1,  │
  │  对象2,  │         │  对象2]      │         │  对象2]      │         │  对象2]  │
  │  对象3]  │         │              │         │              │         │          │
  └──────────┘         └──────────────┘         └──────────────┘         └──────────┘
       │                      │                        │                      │
       │  取出对象1            │  取出对象1              │  取出对象1            │
       │  处理（读取文件）      │  处理（资深作者写意见）   │  处理（读者代表审阅）    │
       │  塞到资深作者队列      │  删除，塞到读者代表队列   │  删除，塞到输出节点队列  │
       │                      │                        │                      │
       ▼                      ▼                        ▼                      ▼
```

**关键规则**：

1. **每个节点有自己的任务队列**。节点不关心全局状态，只看自己的队列里有没有任务。
2. **节点并行运行**。资深作者处理对象1的同时，输入源可以塞对象2进去。读者代表也在同时处理资深作者已经塞过来的任务。
3. **任务在节点间传递**。A 处理完一个任务 → 从自己队列删除 → 塞到所有下游节点的队列尾部。
4. **任务对象在流动中累积文件**。每经过一个节点，节点的产出（报告 + 聊天记录）追加到对象中。
5. **节点不需要知道全局调度**。只需要知道"我的上游是谁，我的下游是谁"，以及"我的队列里有任务就处理"。

### 1.0.1 与集中式调度的对比

| | 集中式调度（错误实现） | 分布式工作节点（正确设计） |
|---|---|---|
| **调度方式** | 一个中央调度器按拓扑层级逐个处理 | 每个节点独立运行，只看自己的队列 |
| **并行性** | 无——同一层级的节点也串行 | 所有节点并行运行 |
| **状态感知** | 调度器需要知道全局状态 | 节点只知道自己队列的状态 |
| **任务传递** | 调度器在层级间移动对象 | 节点完成即塞给下游 |
| **扩展性** | 加节点需要修改调度逻辑 | 加节点只需要定义上下游 |

**当前代码中 ObjectPipelineEngine 是集中式实现——一个对象按层级顺序穿过所有节点，处理完才处理下一个。这需要重构为分布式模型。**

### 1.1 从"文件"到"对象"

当前管道模式中，输入端每个文件是孤立的、无状态的。新架构将每个输入包装为**对象**，对象在整个工作流中流动、累积。

**对象的本质**：

```
对象 = {
  对象名,
  [
    初始文件.md,           // 用户输入
    {专家A意见.md, 聊天内容.json},  // 第一个专家产出
    {专家B意见.md, 聊天内容.json},  // 第二个专家产出
    {专家C意见.md, 聊天内容.json},  // ...
  ]
}
```

- 初始文件来自输入源节点（用户放入的文件）
- 每个专家节点从队列中读取对象，读取其中的所有已有文件
- 处理后将自己的意见报告插入对象，传给下游
- 随着流程推进，对象内的文件列表越来越长

### 1.2 专家升级为 Agent

不再是"收上下文 → 调用 LLM 一次 → 出文本"的固定模式。每个专家节点是一个 **Agent**，接收到对象后：

1. 读取对象内所有文件作为上下文
2. **多轮内部讨论**：生成意见 → 自我审视 → 修正 → 再审视 → ...（多轮迭代）
3. 产出两份文件：
   - `{专家名}意见.md`——最终报告
   - `{专家名}_聊天内容.json`——完整迭代过程

### 1.3 对象流动

```
初始对象：   [初始文件.md]

      ↓ 资深作者 Agent（多轮迭代后）

对象变为：   [初始文件.md, {资深作者意见.md, 聊天内容.json}]

      ↓ 读者代表 Agent（读取全部 3 个文件）

对象变为：   [初始文件.md, {资深作者意见.md, 聊天内容.json},
             {读者代表意见.md, 聊天内容.json}]

      ↓ 剧情架构师 Agent（读取全部 5 个文件）

对象变为：   [初始文件.md, {资深作者意见.md, 聊天内容.json},
             {读者代表意见.md, 聊天内容.json},
             {剧情架构师意见.md, 聊天内容.json}]
```

下游专家可以看到完整的上游讨论历史。

### 3.4 单对象模式 vs ZIP 模式对比

| | 单对象模式 | ZIP 模式 |
|------|------|------|
| **入口** | 输出页面双击对象 | 拖入 ZIP / 导入 ZIP |
| **数据范围** | 单个对象 | ZIP 内所有对象 |
| **右侧布局** | 只有专家列表 | 对象列表（上）+ 专家列表（下） |
| **对象切换** | 无（只有一个对象） | 点击对象列表切换 |
| **适用场景** | 运行刚结束，快速查看 | 复盘历史运行、分享给他人审查 |

### 3.5 两种模式的 URL 路由

```
# 单对象模式
/view?projectId=285db94b&objectId=obj_1_1234567890

# ZIP 模式（导入本地文件后，前端解析）
/view?projectId=285db94b&source=zip

# ZIP 模式（已上传到服务器）
/view?projectId=285db94b&sessionId=2026-05-06_143000
```

### 3.6 与旧"讨论回放"设计的关系

旧设计（时间线 + 迭代弹窗）的思路是"把所有内容平铺在一个时间轴上"。新设计用**对象→专家两层选择**代替，理由：
- 多个对象的 3 个专家全部平铺在一条时间线上，内容太多导致难以导航
- 两层选择（先选对象，再选专家）一次只看一个专家的内容，清晰且性能好
- ZIP 模式下对象+专家的二级导航天然适合树形数据

## 五、Agent 中止条件

用户在工作流的画布中，点击每个专家节点，可以单独配置该专家的**中止方式**。三种模式互不排斥，可同时启用。

### 5.1 模式一：标签中止（`<stop>` 标签）

专家在迭代过程中，若认为讨论已充分，可以在发言中输出 `<stop>` 标签。后端正则检测到该标签后立即终止该专家的迭代，采纳当前发言为最终意见。

```
专家发言：
"...以上就是本轮的完整分析。综合来看卷纲结构合理，无需进一步修正。<stop>"

后端检测到 <stop> → 终止迭代 → 保存当前发言为最终报告
```

**适用场景**：全自动模式、专家对产出质量有信心时快速收敛。

### 5.2 模式二：阻塞中止（用户手动决定）

事先设定一个触发楼层（例如第 3 轮、第 5 轮等）。专家迭代达到此楼层时，流程**阻塞**，界面弹出提示等待用户操作：

```
┌─────────────────────────────────────────────┐
│ 资深作者 已迭代 3 轮                         │
│                                             │
│ 第3轮产出摘要：建议分5卷，担心第3卷节奏...     │
│                                             │
│ [继续迭代] [采纳当前意见并停止] [查看详情]     │
└─────────────────────────────────────────────┘
```

- **继续迭代**：专家再跑一轮
- **采纳当前意见并停止**：取当前发言为最终报告，继续往下游
- **查看详情**：展开完整聊天记录再做决定

**触发楼层配置**：用户在专家节点配置面板中设置"每 N 轮阻塞一次"，0 表示不阻塞。

### 5.3 模式三：楼层中止

设定一个最大楼层数 N。专家迭代满 N 轮后**自动停止**，采纳最后一轮发言为最终意见。

```
节点配置：
  最大迭代轮数: [5]
  超出后行为: [采纳最后一轮] / [回退到最佳轮次]
```

**"回退到最佳轮次"**：当启用自审视机制时，每轮迭代专家会自评质量分数。达到上限后自动选取评分最高的那一轮作为最终报告。

### 5.5 优先级

三种模式同时启用时，触发顺序为：**标签中止 > 阻塞中止 > 楼层中止**。即专家打出 `<stop>` 立刻停，不需要等到阻塞楼层或最大轮数。

## 六、专家对话框（用户与专家直接交流）

### 6.1 定位

每个专家节点的聊天页面中，应始终包含一个对话框，允许用户在 Agent 迭代过程中或结束后**直接与该专家对话**。

与当前聊天弹窗的区别：当前弹窗是**只读的流式显示**——用户看专家发言但无法回复。新增的对话框让用户可以：
- 在专家迭代过程中插话（"方向不对，重点看第三章"）
- 在专家完成后追问（"为什么第一卷建议 50 章而不是 40 章？"）
- 要求专家重新审视某个特定问题

### 6.2 交互流程

```
专家 Agent 正在迭代...
  │
  │  ┌─────────────────────────────┐
  │  │ 第1轮：产出初稿...            │
  │  │ 第2轮：修正中...              │
  │  │                              │
  │  │ ── 用户对话框 ──              │
  │  │ [方向偏了，多关注市场热点]     │
  │  │ [发送]                       │
  │  └─────────────────────────────┘
  │
  ▼
专家收到用户输入 → 结合用户反馈继续迭代
  │
  │  ┌─────────────────────────────┐
  │  │ 第3轮：基于用户反馈调整...     │
  │  │ 产出最终意见                  │
  │  └─────────────────────────────┘
```

### 6.3 与中止条件的配合

用户对话框天然支持阻塞中止场景：
- 达到阻塞楼层时，用户可以直接在对话框中输入修改意见而非仅选"继续/停止"
- 用户插话后，Agent 将用户输入纳入下一轮迭代的上下文

## 七、文件读写标记语言设计

### 7.1 方案：标签包裹，全量写入

每轮迭代，专家用 `<报告>` `</报告>` 标签包裹最终意见，系统提取标签内全部内容覆写文件。

```
AI 本轮输出：
────────────────────────────
思考：4卷还是5卷需要再斟酌...   ← 这部分不进报告

<报告>
# 资深作者意见

## 分卷建议
建议分5卷，每卷约50章...
## 风险提示
第3卷节奏可能偏慢...
</报告>

关于下一轮的方向建议...         ← 标签外的也不进报告
────────────────────────────

系统提取 → 写入 资深作者意见.md（仅 <报告> 标签内内容）
```

- 标签内的内容**完整覆盖**对应意见报告文件
- 标签外的内容（思考、闲聊）只存入聊天 JSON，不进报告
- 正则提取：`/<报告>([\s\S]*?)<\/报告>/`，匹配不到时降级为写入全部 content
- **只扫描 content 部分**，忽略 thinking 中的 `<报告>` 字样（thinking 和 content 已在 SSE 流中分离）

**为什么只需要这一种标签**：全量覆写，不需要行号、段落锚点、追加等复杂语法。`<报告>` 标签唯一职责就是**划定写入边界**。

### 7.2 关于读取：系统自动注入，无需标记

专家**不需要** `[read]` 标记来读取文件。每轮迭代时，系统自动将对象内所有已有文件注入 prompt：

```
## 已有文件
### 01_大纲1.md
{内容...}

### 02_资深作者意见.md
{内容...}

### 02_资深作者_聊天内容.json
{内容...}

---

请基于以上文件发表你的专业意见...
```

选择性读取由节点配置面板控制（已规划在第四节），与标记语言无关。

若未来需要专家"主动拉取"某份文件（非系统自动注入），可增加：
```
[fetch: 读者代表意见.md]
```
表示"我现在需要看这份文件的原文"，系统将其注入后续 prompt。但第一版不需要。

### 7.3 报告与讨论的严格分离

**设计原则**：意见报告和聊天记录是两种完全不同性质的文件，必须分离。

| | 意见报告 (.md) | 聊天记录 (.json) |
|------|------|------|
| **性质** | **交付物**——结构化结论 | **工作草稿**——思考过程的完整录像 |
| **典型内容** | 分卷建议、风险提示、行动方案 | "等等，4卷好像不对……那我改成5卷……不对，还是4卷更合理" |
| **谁写** | 专家 Agent 通过标记语言写入 | 系统自动记录 |
| **写入时机** | 每轮迭代末尾 | 每轮 LLM 调用完成时 |
| **默认读者** | 下游专家（需要干净结论做决策） | 用户复盘、回放页面、调试 |
| **是否注入下游** | 默认注入 | 默认**不注入**（可在节点配置中手动开启） |

**如果不分离会怎样**：

```
下游专家收到的 prompt：
...第1轮：我觉得分4卷 → 不对 → 第2轮：还是5卷 → 等等第3卷可能太拖 → 不对...

下游专家：？？你到底建议分几卷
```

夹杂大量思维垃圾的聊天记录只会干扰下游专家的判断。

**分离后的效果**：

```
下游专家收到的 prompt：
### 资深作者意见.md
建议分5卷，第3卷有节奏风险需增加支线。
原因：...

（干净、直接、可执行）
```

**聊天记录的用途**：
- 回放页面：展示完整的迭代思考过程
- 用户审查：想知道专家为什么最终选了5卷而非4卷，翻聊天记录
- 调试：产出质量差时，追溯是哪一轮思考出了偏差

### 7.4 注入模式 vs AI自读模式

对于被选中读取的文件，用户还可以选择**以什么方式给 AI**：

| 模式 | 行为 | 优点 | 缺点 |
|------|------|------|------|
| **注入提示词** | 系统把文件内容直接拼进 prompt | 简单可靠，AI 无需额外操作 | 占 context，多份文件叠加可能超 token 上限 |
| **AI 自己读** | 系统只注入文件清单（文件名 + 首段摘要），AI 需要时主动 `[fetch: xxx.md]` 拉取 | 省 token，只读真正需要的部分 | 多一轮 LLM 调用，增加延迟 |

**AI 自读模式的工作流程**：

```
系统注入 prompt：
────────────────────────────
## 可用文件
| 文件名 | 摘要 |
|--------|------|
| 资深作者意见.md | 建议分5卷，提醒第3卷节奏风险 |
| 读者代表意见.md | 认为第2卷过渡生硬... |
────────────────────────────

AI 第1轮回复：
"我先看看资深作者的意见中关于节奏的具体分析..."
[fetch: 资深作者意见.md]

系统自动将文件内容追加入上下文 ↓

AI 第2轮回复：
"了解了，资深作者建议在第3卷第45章设高潮点。
我在此基础上建议..."

[section: 剧情架构师意见.md, heading="## 节奏设计"]
...
[/section]
```

**两种模式对比示例**：

```
场景：下游专家需要参考上游 5 位专家的报告（每份 ~2000 字）

注入模式：
  prompt = 5 × 2000 = 10000 字文件内容 + 当前任务
  一次调用搞定，但 10000 字都在 context 里

AI自读模式：
  prompt = 5 × 50 字摘要（共250字）+ 当前任务
  AI 觉得需要看某份 → fetch → 追入该份内容
  可能 2-3 轮交互，但每轮 context 更轻
```

**建议**：
- 报告数量少（1-3份）→ 注入模式，直接全塞
- 报告数量多（5+份）→ AI 自读模式，按需拉取
- 第一版只实现**注入模式**，AI 自读留待后续

### 7.5 节点配置面板（完整）

```
┌─────────────────────────────────────┐
│ 专家节点配置 - 剧情架构师             │
├─────────────────────────────────────┤
│                                     │
│  输入文件                            │
│  ┌───────────────────────────────┐  │
│  │ [☑] 初始输入文件              │  │
│  │ [☑] 上游专家意见报告 (.md)    │  │
│  │ [☐] 上游专家聊天记录 (.json)  │  │
│  │      ⚠ 开启会引入思考噪音     │  │
│  │                                │  │
│  │ 读取方式                       │  │
```

## 八、中断恢复机制

### 8.1 场景

管道运行中，网络波动、LLM 接口超时、服务器重启等导致 SSE 流中断。此时当前轮次的专家输出不完整。

### 8.2 策略：末轮丢弃，重新生成

触发场景分两种：
1. **被动中断**：网络超时、SSE 断开、LLM 报错 → 程序内部抛异常自动触发
2. **用户主动中止**：用户在画布上点击停止 → 同样丢弃未完成的那轮

两种场景行为一致：未输出完的那层楼直接丢弃。

```
中断前：
  第1轮 ✅ 完成
  第2轮 ✅ 完成
  第3轮 ❌ 中断（content 不完整，缺少 closing 标记）

恢复后：
  第1轮 ✅ 保留
  第2轮 ✅ 保留
  第3轮 🗑 丢弃不完整内容 → 🔄 重新生成本楼
```

**中断由程序内部自然检测**：网络超时抛异常、SSE 连接断开、LLM 接口返回错误等。程序捕获异常后自动触发恢复流程。

**判断"不完整"的依据**：
1. 最后一轮的 assistant 消息 `content` 为空或为 null（流已断，内容没写完）
2. 该轮 `completedAt` 为 null（未正常收尾）

### 8.3 实现

```typescript
async function resumeAgent(obj: PipelineObject, expertNode: ExpertNode) {
  const chatLog = loadChatLog(obj, expertNode.expertId);
  
  if (chatLog && chatLog.rounds.length > 0) {
    const lastRound = chatLog.rounds[chatLog.rounds.length - 1];
    const lastMsg = lastRound.messages[lastRound.messages.length - 1];
    
    // 最后一轮不完整 → 丢弃
    if (!isRoundComplete(lastMsg)) {
      chatLog.rounds.pop();
      saveChatLog(obj, expertNode.expertId, chatLog);
    }
  }
  
  // 从中断轮继续迭代
  return expertNode.agent.continueIteration(obj, chatLog);
}

function isRoundComplete(lastMsg: ChatMessage): boolean {
  return lastMsg.role === 'assistant' && lastMsg.content?.length > 0;
}
```

### 8.4 对象级恢复

对象本身持久化为 `manifest.json` + 文件目录，管道重启时扫描未完成的对象，从中断节点继续。

## 九、聊天内容 JSON 存储格式

### 9.1 设计原则

- 按**轮次**（楼层）组织，每轮是多条消息组成的对话
- 区分角色：`system`（自审视提示词）、`assistant`（AI 产出）、`user`（用户插话）
- assistant 消息中 `thinking` 和 `content` 分离
- 对齐现有前端 ChatPopup 和 L1Seed 的消息数据结构

### 9.2 数据结构

```typescript
interface ExpertChatLog {
  // 元数据
  expertId: string;               // 如 "senior_author_v1"
  expertType: string;             // 如 "资深作者"
  objectId: string;               // 所属对象 ID
  reportFile: string;             // 对应的意见报告文件名
  startedAt: string;              // ISO 时间戳
  completedAt: string | null;     // 完成时间
  
  // 轮次列表
  rounds: ExpertRound[];
}

interface ExpertRound {
  round: number;                  // 第几轮（从 1 开始）
  messages: ExpertMessage[];      // 本轮的对话消息
}

interface ExpertMessage {
  role: 'system' | 'assistant' | 'user';
  
  // 以下字段按 role 不同有不同含义
  
  // role === 'system'：系统注入的审视/修正提示词
  // role === 'user'：用户在对话框中输入的内容  
  content?: string;
  
  // role === 'assistant'：AI 的产出
  thinking?: string;              // 思考过程
  content?: string;               // 正文输出
  
  timestamp: string;              // ISO 时间戳
}
```

### 10.3 中间区域：聊天 / 迭代日志

这是页面的核心区域，展示 Agent 的多轮迭代过程。

**消息类型与渲染**：

| SSE 事件 | 前端渲染 |
|----------|---------|
| `agent_round_start` | 轮次分割线 + 轮次编号标签 |
| `agent_chunk` (thinking) | 追加到当前轮次的"思考"折叠面板。默认折叠，用户点击展开 |
| `agent_chunk` (content) | 追加到当前轮次的"产出"区域。流式渲染，逐字显示 |
| `agent_round_complete` | 标记当前轮次完成，显示自评分（如有） |
| `agent_waiting_user` | 阻塞等待提示 + 自动展开输入框 |
| `agent_complete` | 专家完成标记，产出报告自动更新到右侧报告面板 |
| `agent_error` | 红色错误提示 |

**轮次卡片结构**：

```
┌──────────────────────────────────────┐
│ 🔄 第2轮                   12:30:05  │  ← 轮次头：轮号 + 时间
├──────────────────────────────────────┤
│ 💭 思考过程                    [展开] │  ← thinking 折叠区，默认折叠
│ ┌──────────────────────────────┐    │
│ │ 4卷导致第3卷内容过多...        │    │  ← 展开后显示完整思考文本
│ └──────────────────────────────┘    │
│                                      │
│ 📝 产出                              │  ← content 始终显示
│ 维持5卷，将第3卷部分支线移至第2卷...    │  ← 流式写入时实时更新
│                                      │
│ 📊 自评: 7/10                        │  ← 仅在 round_complete 有 score 时显示
└──────────────────────────────────────┘
```

**用户消息气泡**（当用户通过输入框发送消息时）：

```
┌──────────────────────────────────────┐
│ 💬 用户                     12:30:10 │
│ 多关注最近的热点趋势                   │
└──────────────────────────────────────┘
```

用户消息插入到当前迭代流中，与 AI 消息交错排列。

**输入框区域**（固定在中间栏底部）：

```
┌──────────────────────────────────────┐
│ ┌──────────────────────────────┐    │
│ │ 输入反馈或修改意见...          │    │  ← 多行文本框，Enter 发送，Shift+Enter 换行
│ └──────────────────────────────┘    │
│ [发送] [采纳当前产出并停止此专家]       │  ← 两个按钮
└──────────────────────────────────────┘
```

- **发送**：消息追加到聊天流，后端将该消息注入 Agent 下一轮迭代的 `user` message
- **采纳并停止**：发送消息（如有内容），同时发送 `action: 'accept'` 终止该专家迭代

**聊天区域生命周期**：

每个 Agent 迭代期间，聊天区域只展示**当前这一个专家**的讨论过程。当 `agent_complete` 事件到达后，前端做两件事：

1. **归档**：将当前聊天内容（全部轮次的 thinking + content + 用户消息）序列化，追加到对象数据中（即右侧文件队列的聊天 JSON）
2. **清空**：清空中间聊天区域的所有消息，释放 DOM 节点和内存

```
对象处理流程：
  │
  ├─ agent_start (资深作者)
  │   └─ 聊天区：资深作者的迭代日志
  │   └─ agent_complete → 归档 → 清空聊天区
  │
  ├─ agent_start (读者代表)
  │   └─ 聊天区：读者代表的迭代日志（全新，无资深作者残留）
  │   └─ agent_complete → 归档 → 清空聊天区
  │
  └─ agent_start (剧情架构师)
      └─ 聊天区：剧情架构师的迭代日志
      └─ agent_complete → 归档 → 清空聊天区
```

**为什么逐专家清空而非全保留**：

- **性能**：一个专家迭代 5 轮，每轮 thinking + content 可达数千字。三个专家就是 3×5 = 15 轮的长内容。DOM 节点过多导致滚动卡顿、内存飙升
- **关注点**：用户一次只需要看**当前专家**的思考过程，上一个专家的产出已经以报告形式在右侧面板可见
- **回溯**：如需回看任意专家的完整讨论过程，可点击右侧文件队列中的 `{专家名}_聊天内容.json` 展开（或导入回放页面）

**清空时机**：在收到 `agent_complete` 事件并将聊天内容写入对象后立即清空，不等下一个 `agent_start`。

## 十一、群聊设计

### 11.1 定位

**群聊（原"容器"）的本质是多专家同时讨论**。多个专家放入同一个群聊后，不再是各自独立串行处理对象，而是**同时参与一场群聊讨论**。

与单专家节点的区别：

| | 单专家节点 | 群聊 |
|------|------|------|
| **参与者** | 1 个专家 | 多个专家 |
| **讨论方式** | Agent 自迭代（多轮自我审视） | 专家之间互相 @、回复 |
| **对象** | 处理后追加产出，传给下游 | 群聊内所有专家共享同一个对象 |
| **退出条件** | 标签 / 阻塞 / 楼层 | 手动 / 共识 / 多数赞同 / 门禁专家 |
| **产出** | 每个专家的报告 + 聊天记录 | 同上，但聊天记录是群聊记录而非单人 |

### 11.2 拓扑中位置

群聊在 DAG 拓扑中是一个节点。群聊内的专家不单独出现在拓扑中。

```
输入源 ──→ [资深作者] ──→ [┌─────────────┐] ──→ 输出
                          │ 群聊：审核组    │
                          │ 读者代表        │
                          │ 网络编辑        │
                          │ 人物设计师      │
                          └─────────────┘
```

- 对象先经资深作者单独处理
- 然后进入"审核组"群聊，三位专家同时讨论
- 群聊结束后，三位专家的产出一起追加到对象，对象流向输出

### 11.3 两种发言模式

**顺序循环（ordered）**：群聊内专家按连线顺序依次发言。每轮每人发言一次。适合正式评审流程。

```
第1轮：读者代表 → 网络编辑 → 人物设计师
第2轮：读者代表 → 网络编辑 → 人物设计师
...直到退出条件满足
```

**提及驱动（mention_driven）**：群聊内专家可以 @ 其他人，被 @ 的人接话。发言顺序由讨论内容动态决定。适合自由讨论。

```
读者代表：整体节奏有问题。@网络编辑 你觉得呢？
网络编辑：确实，前三章太慢。@人物设计师 角色弧线需要调整吗？
人物设计师：主角的成长线太模糊了，建议在第二章加一个关键事件...
读者代表：同意。@网络编辑 从市场角度看呢？
```

### 11.4 退出条件

群聊配置面板中的退出条件：

| 模式 | 行为 |
|------|------|
| **手动** | 用户点击"停止"结束群聊 |
| **全部赞同** | 所有专家打出 `<stop>` 或明确表示同意后退出 |
| **多数赞同** | 超过 `exitRatio` 比例的专家同意后退出 |
| **门禁专家** | 指定一个 gatekeeper，该专家认为可以定稿后退出 |

### 11.5 上下文注入

群聊内的专家在发言时，prompt 中注入：

1. **对象已有文件**（同单专家节点）
2. **本轮群聊历史**：群聊内其他专家**在本轮**的发言
3. **历史轮次摘要**：为防止 token 爆炸，上一轮的详细讨论在下一轮中压缩为摘要

```
## 群聊上下文

### 当前对象
[对象内所有文件...]

### 本轮讨论
[读者代表]：整体节奏有问题...
[网络编辑]：前三章太慢...

### 上轮摘要
第1轮讨论了节奏和角色问题，读者代表和网络编辑认为前三章太慢，
人物设计师建议在第二章增加关键事件。共识方向：加快开头节奏。
```

**上下文深度控制**（群聊配置面板）：

群聊越长，历史越多。用户可限制注入的上下文量：

| 配置项 | 行为 | 适用场景 |
|--------|------|---------|
| **按楼层** | 只读最近 N 层发言（一层 = 群聊内所有专家各说一次） | 控制逻辑连贯性 |
| **按 token** | 约 N token 的最近历史 | 控制成本 |
| **两者都启用** | 以先触达的为准 | 双重保险 |
| **都不启用** | 全量历史 | 讨论较短时 |

### 11.6 I/O 引脚

群聊对外暴露**一个输入端口**和**一个输出端口**，类似电路设计中的"模块封装"：

```
                    ┌─── 群聊: 审核组 ──────────────┐
                    │                                │
  上游对象 ──────→ 输入                              │
                    │  ┌──────┐   ┌──────┐          │
                    │  │读者代表│──→│网络编辑│          │
                    │  └──────┘   └──────┘          │
                    │      │          │              │
                    │      └────┬─────┘              │
                    │           ▼                    │
                    │  ┌────────────┐               │
                    │  │ 人物设计师  │               │
                    │  └────────────┘               │
                    │                                │
                    └──── 输出 ────────────→ 下游节点
                         群聊全文 + 各专家报告
```

- **输入**：接收上游节点的对象（含累积文件）
- **输出**：对象 + 群聊内所有专家的报告 + 群聊 JSON 日志
- 群聊对外是黑盒，下游只看到"审核组产出了什么"，不关心里面怎么讨论的

**关于多 I/O**：从逻辑上讲，群聊和专家节点都可以有多个输入和多个输出（接收多方数据、产出多种文件）。但多 I/O 大幅增加连线管理和数据路由的复杂度。**第一版只保留单输入单输出**，多 I/O 留待日后扩展。

### 11.7 世界书 / RAG 绑定

群聊和专家节点可绑定世界书和 RAG 实例（实例管理见笔记17 第九节，待后续实现）：

```
群聊配置:
  ┌─────────────────────────┐
  │ 世界书绑定: [核心设定 ▾]  │
  │            [+添加]       │
  │ RAG绑定:   [历史回顾 ▾]  │
  │            [+添加]       │
  └─────────────────────────┘
```

绑定后在群聊发言前自动注入相关条目。例如人物设计师绑了"角色追踪"世界书，发言前自动查询角色相关信息。

### 11.8 实现方式

群聊在后端实现为一个特殊的处理节点：

```
ObjectPipelineEngine.processLevel():
  for each node in level:
    if node is groupChat:
      await processGroupChat(node, object)
    else:
      await processSingleExpert(node, object)
```

`processGroupChat` 内部是一个循环：
1. 按发言模式确定下一个发言的专家
2. 为该专家构建上下文（对象文件 + 群聊历史）
3. 调用 `expert.speakStream()`（单次发言，非 Agent 迭代）
4. 将发言追加到群聊日志
5. 检查退出条件
6. 不满足则继续下一轮

**与单专家 Agent 迭代的区别**：群聊中每个专家调用 `speakStream()`（单次 LLM 调用），而非 `iterate()`（多轮自我审视）。群聊的"迭代"是专家之间的来回对话，而非个人内部反思。

## 十二、总结

七个核心变化：

1. **输入：文件 → 对象** —— 文件有归属，状态可累积
2. **专家：单次发言 → Agent 迭代** —— 多轮讨论 + 自我修正，产出意见报告 + 聊天记录
3. **群聊** —— 多个专家同时讨论，顺序循环或 @ 提及驱动，有退出条件。单输入单输出，多 I/O 日后扩展
4. **中止：固定轮次 → 三种可配模式** —— 标签中止 / 阻塞中止 / 楼层中止，每个专家节点独立配置
5. **交互：只读旁观 → 可对话专家** —— 每个专家聊天页面增加对话框，用户可直接与专家交流、插话指导
6. **输出：散文件 → ZIP 打包 + 回放页面** —— 可归档、可复盘、可分享
7. **UI：三栏布局** —— 文档库（左）+ 迭代日志+输入框+产出文档（中）+ 文件队列+报告/群聊成员（右）
