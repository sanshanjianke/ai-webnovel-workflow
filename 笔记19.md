# 笔记19：标签映射层（L3/L4）架构设计

## 一、背景

L3（叙事层）和 L4（渲染层）目前已有独立页面和路由，但功能较简——默认技能，标签映射规则靠 prompt 内嵌，无自定义能力。

核心问题：L3/L4 该不该塞进编排层？

最初结论是"不该"——因为把 L3/L4 理解为"单人工具箱"。但这个理解是错的。标签插入本身就是多专家协作的天然场景：同一段草稿，情节标签、人物标签、叙事标签、爽点标签需要不同专业的"眼睛"来看。这和 L2 多专家讨论大纲是同构的。

**最终结论：直接在编排层实现，复用现有引擎。**

---

## 二、核心洞察

标签不是一个人闷头贴的。一段草稿需要：

| 专家 | 关注 | 插入的标签类型 |
|------|------|--------------|
| **情节解析师** | 功能序列是否完整 | `【功能】``【序列】``【转折点】` 等 |
| **人物解析师** | 行动元分配、人设一致 | `【人物】``【行动元】``【原型】` 等 |
| **叙事解析师** | 视角、节奏、话语模式 | `【视角】``【节奏】``【话语模式】` 等 |
| **爽点解析师** | 读者情绪曲线 | `【爽点】``【期待感】``【压抑-释放】` 等 |
| **编剧总结师** | 全局审视、增删改标签、给出评价 | 审核 + 修改 + 总结 |

每个专家带着自己的标签库（JSON），读同一段草稿，从自己的专业角度插标签。这和 L2 的"资深作者 + 读者代表 + 剧情架构师共审大纲"完全同构。

---

## 三、架构：直接复用编排层

**不需要新建引擎。** L3 标签专家 = 特殊类型的 Expert，L4 渲染 = 另一个 Expert 或技能节点。

| 编排层（L2） | 标签层（L3/L4） |
|---|---|
| 专家 JSON（`experts/`） | 同样的专家 JSON，额外绑定标签文件 |
| ExpertNode | 复用 |
| GroupNode（群聊） | 复用——标签专家可以群聊讨论 |
| OrchestrationCanvas | 复用——同一个画布，不同类型的工作流 |
| pipeline-workers.ts | 复用——四个 Worker 类型不变 |
| ChatPopup / GroupChatView | 复用 |
| 对象 = 大纲片段 | 对象 = 章节草稿（附标签） |
| 世界书 / RAG | 标签库作为一种新的注入源 |

画布上用户可以创建两种工作流：
1. **大纲工作流**（L2）：资深作者 → 读者代表 → 剧情架构师 → ...
2. **标签工作流**（L3）：情节解析师 → 人物解析师 → 叙事解析师 → 爽点解析师 → 编剧总结师 → ...
3. **混合工作流**：甚至可以把两者串在一起——大纲产出后直接进入标签管道

## 四、标签系统

### 4.1 标签管理页面（TagManager.vue）

独立的标签管理页面，类似世界书管理页：

- 表格展示所有标签，按**双轴分类**（类型 × 流派）：
  - **类型轴**：情节 / 人物 / 叙事 / 爽点
  - **流派轴**：无限流 / 穿越流 / 重生流 / 历史流 / 系统文 / 种田文 / 等等（用户可自定义流派标签）
- 每行：效果标签名、叙事指令（视角/节奏/话语模式）、理由、关联网文概念
- 增删改、批量导入/导出 JSON
- 默认内置 50-100 条核心映射
- 用户完全自由编写标签，无格式校验。写得好的标签可导出为模板供其他用户交流

### 4.2 标签 JSON 格式

```json
{
  "tagId": "装逼打脸_压抑阶段",
  "category": "爽点",
  "genre": ["无限流", "系统文"],
  "效果标签": "装逼打脸-压抑阶段",
  "叙事指令": {
    "视角": "内聚焦（反派/路人视角）",
    "节奏": "慢速叙事（扩述/场景）",
    "话语模式": "大量对话+环境描写"
  },
  "理由": "通过无知者的傲慢视角，制造信息差",
  "关联网文概念": ["欲扬先抑", "期待感悬置", "信息差"],
  "适用专家": ["爽点解析师", "叙事解析师"]
}
```

`genre` 字段用于流派筛选——同一标签可用于多个流派，用户可在 TagManager 中按流派过滤标签视图。

### 4.3 标签目录结构

```
backend-node/tags/
├── 情节/
│   ├── 功能标签.json
│   ├── 序列标签.json
│   └── 情节结构.json
├── 人物/
│   ├── 行动元标签.json
│   ├── 人物原型.json
│   └── 人物关系.json
├── 叙事/
│   ├── 视角标签.json
│   ├── 节奏标签.json
│   └── 话语模式.json
├── 爽点/
│   ├── 装逼打脸.json
│   ├── 悬念铺设.json
│   ├── 战斗场面.json
│   └── 情感爆发.json
└── 用户自定义/
    ├── 我的标签.json          # 用户随意编写，无格式约束
    └── 作者社区模板.json       # 导入他人分享的标签模板
```

标签本身带 `category`（类型轴）和 `genre`（流派轴）字段，不依赖目录结构分类。TagManager 中按双轴交叉过滤。用户自定义标签完全自由，写得好的可导出交流。

### 4.4 标签专家 JSON 定义

每个标签专家声明自己绑定哪些标签文件：

```json
{
  "expertId": "plot_tagger_v1",
  "expertType": "情节解析师",
  "icon": "🏗",
  "description": "分析章节草稿中的功能序列、情节结构，插入情节相关标签",
  "temperature": 0.5,
  "tagBindings": ["情节/功能标签.json", "情节/序列标签.json", "情节/情节结构.json"],
  "prompt_template": "你是一位情节解析师。根据以下标签库中的标签定义，在草稿中插入适当的情节标签...",
  "role_instructions": {
    "main": "你是情节解析师，负责在草稿中标注功能、序列、情节结构。",
    "review": "审核情节标签的完整性和准确性。",
    "supplement": "补充遗漏的情节标签。"
  },
  "agent_defaults": {
    "enableTagStop": true,
    "maxRounds": 2,
    "onMaxRounds": "accept_last"
  }
}
```

`tagBindings` 在专家发言前自动加载——系统读取 JSON 文件内容，将标签定义注入专家的 prompt 上下文，类似世界书的触发词注入。

### 4.5 标签版本管理

用户修改标签 JSON 后，旧草稿中的标签面临版本不一致问题。策略：**能替换就替换，替换不了就保留**。

- 系统记录每个标签的版本号，存储在标签 JSON 的 `version` 字段中
- 打开旧草稿时，检测其中标签的版本和当前标签库中的版本是否一致
- 能自动替换的：新旧标签的标签名相同、属性键相同，只是属性值或描述文字变化 → 自动更新
- 不能自动替换的：标签名变了、属性结构变了 → 保留旧标签。对 L4 来说，读到一个旧版本的标签也比什么都没有好
- 用户可手动在 TagManager 中触发"批量更新所有草稿中的标签 X"

不需要世界书那种复杂的版本链——标签是静态知识库，不像世界书那样随剧情持续变化。简单的"能换就换，不能换就留着"即可。

## 五、典型工作流

用户在大纲工作流产出精修大纲后，切换到标签工作流：

```
输入源（章纲+草稿）
    │
    ▼
情节解析师──→ 人物解析师──→ 叙事解析师──→ 爽点解析师
    │              │              │              │
    │  插入功能     │  插入行动元   │  插入视角     │  插入爽点
    │  序列标签     │  原型标签     │  节奏标签     │  情绪标签
    │              │              │              │
    └──────────────┴──────────────┴──────────────┘
                           │
                    ┌──────┴──────┐
                    ▼             ▼
              编剧总结师      ┌─────────────┐
              （审核+增删改）   │ 群聊：争议评审 │
                    │         │ 情节+人物+叙事 │
                    │         │ 讨论冲突标签    │
                    │         └─────────────┘
                    ▼
              输出节点（带标签草稿）
                    │
                    ▼
              L4 渲染（标签展开→正文）
```

标签专家可以**串行管道**（逐个插入不同维度的标签），也可以**并行群聊**（遇到争议时拉群讨论）。这和 L2 的 ExpertNode + GroupNode 一模一样的逻辑。

**串行优先**：第一阶段只实现串行管道。并行模式下，草稿同时发给情节解析师和人物解析师，两边各自产出独立的带标签草稿（两份文档），由专门的后处理程序合并成一份——不靠锁机制。并行合并后延实施。

## 六、标签语法设计

### 6.1 为什么不用 `<>`

最初设计沿用 HTML/XML 风格的尖括号标签（`<视角 type="外聚焦">`），但存在一个问题——**分布偏置**。

**三层原因**：

1. **训练分布偏置**（dense 和 MoE 都受影响）：`<>` 在训练数据中压倒性地出现在代码中（HTML/XML/JSX）。模型见过的 `<>` token 序列，后面接的几乎总是 `div`、`html`、`component`，而不是 `视角`、`爽点`。草稿中大量 `<>` 会隐式地把生成偏向推向代码领域的分布，和"你是X专家"是同一个原理——只不过方向是反的。

2. **MoE 路由放大**：在 dense 模型中这只是概率偏置；在 MoE 中，携带 `<>` 的 token 可能直接被路由器分配给偏向编程的 expert，变成确定性路由错误。

3. **上下文框架效应**：一段文学草稿嵌着 HTML 风格的标签，对模型来说是一个矛盾信号——它不确定该按"写小说"还是"写代码"来继续。

> 补充：prompt 工程中"你是嵌入式设计师/高级程序员"的模式早在 MoE 之前就有效，说明即使 dense 模型，内部表示空间也存在功能分区——不同领域的训练模式形成了不同的"引力区"。prompt 中的身份描述把模型拉向对应区域。`【】` 和 `<>` 的选择同理。

### 6.2 方案对比

| 方案 | 语法 | 优点 | 缺点 |
|------|------|------|------|
| A（选定） | `【视角:外聚焦】...【/视角】` | MoE 安全，中文读者自然理解为标注，视觉突出 | 占 2 个中文字符宽度 |
| B | `> 视角 外聚焦` | Markdown 原生，纯自然语言格式 | 和正文混排时视觉混淆 |
| C | JSON 分离（不混排） | 零路由干扰 | 人不可读，用户无法直接在编辑器中看到标签位置 |

### 6.3 语法规范

标签使用中文方括号 `【】`，采用**自闭和双标签两种模式**：

**自闭和**（单标签，用于标注点）：

```
【功能:转折】【爽点:满足感 强度:↑】【视角:外聚焦】
```

**包围标签**（用于标注段落范围）：

```
【视角:外聚焦】
李昊走进拍卖场时，所有人都在盯着他手里的黑色晶核。
鉴定师老周接过晶核，仪器数值瞬间爆表——165000。全场哗然。
【/视角】
```

**属性语法**：`【标签名:值 属性1:值1 属性2:值2】`，空格分隔多个属性。

标签类型和短横线连接子类型：

```
【爽点:装逼打脸-爆发 stage:高潮】【功能:铺垫-阻碍】【视角:外聚焦】
```

**嵌套标签的处理**：草稿中允许嵌套标签（如 `【视角:外聚焦】【功能:转折】...正文...【/功能】【/视角】`），但在 L3 处理完毕后，同一位置的标签合并为一个：`【标签：A、B】正文【/标签】`。L3 输出时所有标签已经被解析和合并，不再有嵌套结构。L4 接收到的就是干净文本。

### 6.4 标签位置：行内 vs 尾注

标签按内容和长度分为两类，插入位置不同：

**行内标签**：嵌在草稿对应位置，内容是短约束指令（一行以内）。L4 渲染到该位置时遵守即可，不影响后续。

| 类别 | 示例 | 展开后 |
|------|------|--------|
| 视角 | `【视角:外聚焦】` | 约 1 句约束 |
| 节奏 | `【节奏:慢速扩述】` | 约 1 句约束 |
| 功能 | `【功能:转折】` | 约 1 句约束 |

**尾注标签**：统一放在正文末尾，带长参考文本或风格范例。这些标签如果嵌在草稿中间会把模型注意力拉偏——"参考范例"的文本强势程度远高于"约束指令"，模型容易开始模仿范例而非继续写故事。

| 类别 | 示例 | 展开后 |
|------|------|--------|
| 风格 | `【风格:古龙】` | ~200 字风格参考文本 + 范例片段 |
| 文风 | `【文风:冷硬】` | ~150 字文风描述 + 句式示例 |
| 对话风格 | `【对话:海明威式】` | ~100 字范例对话 |

尾注标签在 prompt 中的形态：

```
## 正文草稿
[带行内标签的正文...]

## 风格参考
以下风格指令在正文末尾生效，请整体遵循而非逐字模仿：
- 【风格:古龙】参考文本：短句密集、动作>心理、环境氛围浓、留白多...
- 【对话:海明威式】参考文本：极简对话、大量潜台词、回避直接表述...
```

这样 L4 渲染时知道"整体用这个风格"，而不是在某个具体位置突然切换。

### 6.5 标签处理：全部在 L3 完成

**核心原则：L3 输出经过完整处理的文本，L4 只做文本生成。** L4 的提示词越少、和正文无关的内容越少，生成质量越高——AI 的注意力是有限的。

因此标签展开不放在 L3→L4 过渡阶段，而是放在 L3 管道内部——编剧总结师之后、输出节点之前，增加一个**标签解析步骤**：

```
L3 草稿（含嵌套【】标签 + 正文）
     │
     ▼
编剧总结师审核完毕
     │
     ▼
标签解析器（系统执行，不经过 LLM）
  ├── 正则匹配所有 【标签名 属性...】...【/标签名】
  ├── 嵌套标签合并为单一标签：【标签：A、B】正文【/标签】
  ├── 从映射表查对应完整提示词，注入属性
  ├── 行内标签展开为简短约束句子，嵌在对应位置
  └── 尾注标签（风格类）提取到 prompt 前缀，不留在正文中
     │
     ▼
输出节点：已处理草稿（标签已展开为约束文本）
     │
     ▼
L4 渲染（只看到干净的约束文本 + 正文草稿，无【】标签语法）
```

示例：

```
L3 原始:   【视角:外聚焦】正文【/视角】
标签解析:   [此处使用外聚焦视角：只写外部可见的行为和对话，不进入任何人物的内心。]
L4 看到:   [此处使用外聚焦视角：只写外部可见的行为和对话...] 正文
```

标签在进入 L4 之前已完全解析为自然语言约束。L4 不需要理解标签语法，不需要查映射表，只需要遵守自然语言指令生成正文。

尾注标签（风格参考）更是如此——`【风格:古龙】` 展开为 200 字风格参考后，必须放入 L4 的 system prompt 前缀，而非混在正文中。L4 从一开始就带着风格约束写作。

### 6.6 标签在 UI 中的展示

标签在编辑器中以内联色块显示，不同类别不同颜色：

| 类别 | 颜色 | 示例 |
|------|------|------|
| 情节 | 蓝底 | `【功能:转折】` |
| 人物 | 绿底 | `【行动元:主体】` |
| 叙事 | 紫底 | `【视角:外聚焦】` |
| 爽点 | 橙底 | `【爽点:装逼打脸-爆发】` |

点击色块可以展开属性详情面板，查看和编辑属性。

## 七、L4 渲染层

**L4 的核心定位：只做文本生成。** L4 是特殊的——提示词越多、和正文无关的内容越多，越容易影响生成质量，因为 AI 的注意力是有限的。所有标签处理、约束解析、风格指令展开都在 L3 完成，L4 拿到的就是干净的约束文本。

### 7.1 技能管道

L4 技能 = 特殊类型的 Expert，每种技能一个 JSON 定义，放在标签工作流的末端。7 种技能可串联为渲染管道：

```
带标签草稿 → 初稿扩展 → 描写增强 → 对话打磨 → 终稿润色 → 精修正文
              │           │          │          │
              │ 扩展字数   │ 丰富细节   │ 优化对话   │ 检查打磨
              │ 遵守约束   │ 感官描写   │ 消除生硬   │ 错别字修复
```

用户可以从 7 种技能中选择并编排管道顺序，不需要全部使用。

```json
{
  "expertId": "初稿扩展",
  "expertType": "初稿扩展",
  "icon": "✍️",
  "description": "接收标签草稿，扩展打磨成精修正文",
  "prompt_template": "你是一位执行作家。请严格遵循草稿中嵌入的标签指令...",
  "agent_defaults": {
    "maxRounds": 1,
    "onMaxRounds": "accept_last"
  }
}
```

### 7.2 打回重做节点

L4 渲染后的正文可能需要重新过 L3 标签管道（标签不足、标签矛盾、渲染偏离预期等）。为此增加一个**判断节点**，有两个输出端口：

```
                      ┌──→ 输出（通过，正文合格）
                      │
L4 渲染技能链 → 判断节点
                      │
                      └──→ 打回 L3 标签管道（重标标签后重新渲染）
```

判断节点的决策来源：
- **自动**：检测标签约束是否被遵守（视角/节奏/字数等），偏离超过阈值则打回
- **专家审核**：编剧总结师或专门的审稿专家评估正文质量，决定通过或打回
- **用户手动**：用户在聊天窗口预览正文后手动选择"通过"或"打回"

### 7.3 L3 的输入来源

L2 编排层产出的是**以章为单位的精修大纲 ZIP 包**。用户将 ZIP 拖入文档库，L3 需要时再从文档库拖到输入源节点。

两层之间不通过 DAG 连线直接对接——靠文档库作为中间存储。这样两个工作流在画布上各自独立，互不依赖运行状态：

```
L2 管道完成 → ZIP 导出 → 拖入文档库 → 拖入 L3 输入源 → L3 管道启动
```

输入源节点解析 ZIP 后，每个章纲作为一个任务对象，逐个进入标签管道。每个章纲对象处理时可附带：
- 前后各 N 章作为上下文参考（可配置章数）
- 世界书相关条目（当前人物状态、活跃伏笔等）
- RAG 检索结果（历史章节中类似场景的处理方式）

### 7.4 标签冲突处理

不同专家在同一位置插入矛盾标签时，流程如下：

**第一步——编剧总结师自动处理**：编剧总结师在审核阶段检测标签冲突，能自动判断的情况直接决定保留哪个。判断依据：
- 冲突标签的类型优先级（如爽点优先级高于节奏——"爆发"场景中即使标了"慢速扩述"也应服从快节奏）
- 标签的适用范围（包围标签优先于自闭和标签）
- 上下文（前后段落的标签趋势）

**第二步——阻塞等用户**：编剧总结师无法判断时（两个标签都有道理），标记为冲突状态，发通知给用户。用户在 TagManager 或聊天窗口查看冲突位置、两个标签的理由，手动选择保留哪个。

**第三步——都保留**：如果两个冲突标签各有道理且不互斥，就都留下。后面的专家（L4 渲染技能或更下游的专家）自行判断采用哪个。宁可多标不错标——L4 即使只读到一个标签也比没有好。

## 十一、待讨论问题

以下问题是审查设计后发现的——不是方案本身的缺陷，而是设计尚未覆盖的细节，需要在实施前明确。

### 11.1 标签注入的 token 成本（已决策）

每个标签专家绑定若干标签 JSON 文件。即使标签库总量大，但：

- 用户可自定义标签专家绑定哪些标签文件（和 L2 自定义专家一样），不需要全量注入
- 即使绑定全部标签，一章实际用上的也就四五个/专家，五个专家加审核后总计约二十来个标签
- 标签专家的 prompt 里列出全部可用标签定义，让专家自己从库中选择适合当前草稿的标签——这和世界书的触发词注入不是同一个模式（标签库是"菜单"，不是"自动匹配"）

**决策**：不做预筛选。标签专家看到绑定文件中的全部标签，自行判断用哪个。

### 11.2 并行专家合并草稿版本（已决策）

并行模式下，同一份草稿同时发给情节解析师和人物解析师。两边各自产出独立的带标签草稿（两份文档），由专门的后处理程序合并成一份。不靠锁机制。

**决策**：第一阶段只实现串行管道。并行生成 + 后处理合并的方案后延。

### 11.3 包围标签的重叠与嵌套（已决策）

`【视角:外聚焦】...【/视角】` 这种包围标签覆盖一段文本范围。如果情节解析师在段落 1-3 上标了 `【功能:转折】`，人物解析师在段落 2-4 上标了 `【行动元:主体】`，产生的是交叉重叠而非简单嵌套。

**问题**：
- 包围标签的边界是"语义范围"还是"精确字节位置"？
- LLM 如何准确输出包围标签的闭合位置？它可能在"差不多"的位置就加 `【/标签】`，导致范围不精确
- 两个重叠但不包含的包围标签，在 L4 渲染时如何处理？内层约束先执行还是外层的？

建议约束：包围标签只允许嵌套（完全包含）或并列（不重叠），禁止交叉。交叉场景由编剧总结师检测并手动修正。

### 11.4 标签插入的副作用：LLM 可能修改原文（已决策）

标签专家需要输出"草稿+插入的标签"，LLM 可能会不自觉地修改草稿原文。

**决策**：不搞复杂的 diff/行号方案。两件事就够了——
1. **prompt 中明确要求**：禁止修改原文措辞，只允许插入标签
2. **输出后算法检测**：比较专家输出和原始草稿的文本相似度，偏差过大则打回重改

### 11.5 打回重做节点的引擎改动（设计中）

打回重做节点（§7.2）需要条件路由——节点根据审核结果选择"通过"或"打回"两个输出方向。当前引擎只支持单输出 DAG，需要重构以支持**多端口 + 条件边**。

**策略**：先在 L3/L4 实现并实验，验证稳定后迁移到 L2。L2 同样需要这个能力（专家审核产出、打回重做）。

**设计方案：端口化边模型**

核心改动：

1. **边增加 `sourcePort` 字段**：

```typescript
// protocols/index.ts
export interface EdgeConfig {
  source: string;
  target: string;
  type?: string;       // 'data_flow' | 'pass' | 'reject'
  sourcePort?: string; // 匹配节点的输出端口名，默认 'default'
}
```

2. **节点可声明多输出端口**（新增类型）：

```typescript
export interface OutputPort {
  name: string;     // 'pass' | 'reject' | 'default'
  label: string;    // 前端显示名
  color?: string;   // 连线颜色
}

// NodeConfig 增加字段：
export interface NodeConfig {
  // ... existing fields ...
  outputPorts?: OutputPort[];  // 多端口节点声明，默认 [{name:'default', label:'输出'}]
}
```

3. **运行时路由**：节点 Worker 处理完对象后，在对象上设 `routePort`。引擎分发时只走匹配 `sourcePort` 的边：

```typescript
// object-pipeline-engine.ts 分发逻辑
const outPort = obj.meta.routePort || 'default';
for (const [downId, queue] of downQueues) {
  const edge = edges.find(e => e.source === nodeId && e.target === downId);
  const edgePort = edge?.sourcePort || 'default';
  if (edgePort === outPort) {
    queue.enqueue(obj);
  }
}
```

4. **循环保护**：PipelineObject 上维护 `rejectCount`，打回时递增。超出阈值（默认 3）强行终止，对象走 `pass` 端口到输出节点。

```typescript
// PipelineObject meta 增加：
interface PipelineObjectMeta {
  rejectCount: number;   // 打回次数
  maxRejects: number;    // 上限，默认 3
  routePort?: string;    // 当前路由方向
}
```

5. **判断节点类型**（新增 `judgment` 节点类型）：不是 Expert 也不是 Container，而是轻量判断——调用 LLM 做一次简单的"通过/打回"评估，设置 `routePort`。用户也可手动选择。

**实施路径**：

```
Phase 1：引擎层
├── EdgeConfig 加 sourcePort
├── NodeConfig 加 outputPorts
├── object-pipeline-engine.ts 路由逻辑改为端口匹配
├── PipelineObject 加 rejectCount
└── 前端 OrchestrationCanvas 连线面板支持端口选择

Phase 2：判断节点
├── JudgmentNode.vue（或复用 ExpertNode.vue 的新 type）
├── 判断节点 Worker（轻量 LLM 调用，结构化输出 pass/reject + 理由）
└── 手动判断模式（用户直接在聊天窗口点"通过"或"打回"）

Phase 3：实验 → 迁移
├── L3/L4 验证稳定性
└── 迁移到 L2 编排层
```

**向后兼容**：现有节点不走 `sourcePort`，引擎用 `'default'` 兜底，行为不变。

**前端连线 UX**：打回重做节点有两个可视端口（左侧通过、右侧打回），连线时从不同端口拖出。打回连线用红色虚线，通过连线用绿色实线，视觉上明确区分。

### 11.6 尾注标签与 L4 流式渲染的时序矛盾（已决策）

尾注标签（§6.4）放在正文末尾，但 L4 渲染是**流式的**——从第一句开始逐段生成。

**决策**：尾注标签在 L3 标签解析步骤中提前提取并展开，放入 L4 的 system prompt 前缀。正则扫描草稿 → 提取所有尾注标签 → 展开为风格约束 → 放入 prompt 前缀。L4 从一开始就带着风格约束生成，正文中不再出现尾注标签。

### 11.7 标签库的"适用专家"与专家的"tagBindings"双向过滤

标签 JSON（§4.2）中有 `适用专家` 字段声明"哪些专家应该用这个标签"，专家 JSON（§4.4）中有 `tagBindings` 声明"我绑定哪些标签文件"。这是两套独立的过滤逻辑。

**问题**：如果标签 X 的 `适用专家` 包含"情节解析师"，但情节解析师的 `tagBindings` 没有包含标签 X 所在的文件，这个标签用不用？

应该明确：`tagBindings` 是主键——专家读哪些标签文件，就从文件中取全部标签。`适用专家` 字段降级为"推荐提示"而非硬约束，用于前端 TagManager 显示"这个标签推荐给哪些专家"。

### 11.8 "标签"一词的二义性（已解决）

`<stop>` 相关已全部从 `tag` 重命名为 `stop`（2026-05-08）：

- `enableTagStop` → `enableStopSignal`（protocols + base.ts + meeting.ts + pipeline-workers.ts + 7 个 expert JSON）
- `hasStopTag` → `hasStopSignal`（base.ts）
- `stoppedBy: 'tag'` → `stoppedBy: 'stop'`（base.ts）
- 前端 `agentTagStop` → `agentStopSignal`，"标签中止" → "`<stop>` 中止"（OrchestrationCanvas.vue）

`【】` 叙事标注独占 `tag` 命名空间（`tagBindings`、`TagManager`、`tags/` 目录），不再有二义性。

### 11.9 标签库的初始规模与维护成本（已决策）

**决策**：标签库不做系统级验证。用户想怎么写就怎么写。写得好的标签可以导出为模板供其他用户交流。TagManager 不做视角枚举校验、不做叙事指令合理性检查——标签的效果由用户自己判断。

### 11.10 与现有 L3/L4 代码的关系（已完成）

旧 L3/L4 独立页面和后端路由已废弃删除（2026-05-08）：

- **已删除**：`L3Narrative.vue`、`L4Render.vue`、`l3.ts`、`l4.ts`
- **已清理**：12 个文件中的 import/路由/导航/类型/配置引用（main.js、App.vue、Dashboard、Library、DocumentSidebar、Settings、routes/index.ts、settings.ts、config.ts、protocols/index.ts、library.ts、代码框架.md）
- **旧数据**：`l3_chapter_plan.json` 和 `l4_text.json` 的迁移逻辑保留在 `library.ts` 中（用于旧项目数据兼容），但不再生成新文件
